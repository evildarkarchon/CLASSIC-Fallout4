- [x] 11.1 Create `.github/workflows/publish-yaml-data.yml` triggered only by `push: tags: ['yaml-data-v*']`
- [x] 11.2 Validation job: Python step (reusing `tools/` conventions) that parses every file in `CLASSIC Data/databases/*.yaml` and asserts `schema_version` exists and matches `^\d+\.\d+$`; fails the workflow on any violation — `tools/publish_yaml_data/validate.py` uses `ruamel.yaml` with `pure=True` because the libyaml-backed C parser rejects bare-scalar `foo::bar` flow-sequence entries (e.g. `stack_contains_any: [usvfs::hook_DeleteFileW, ::Manager]`) that the runtime Rust `yaml-rust2` parser accepts. - [x] 11.4 Manifest job: generate `manifest.json` from the validated files (schema_version + sha256 + size_bytes + release_tag + published_at); every `files[].download_url` is constructed from `https://github.com/<owner>/<repo>/releases/download/<tag>/` + URL-encoded asset name; per-file `min_client_schema` / `max_client_schema` read from an in-repo metadata file (e.g., `CLASSIC Data/databases/client-schema-ranges.yaml`) — `urllib.parse.quote(name, safe="")` so spaces become `%20` (not `+`), which is what the release-asset CDN expects. `manifest.signatures` is always emitted as `[]` by this change; a follow-up `yaml-update-sigstore-verify` change owns populating it with a signed descriptor.
- [x] 11.5 Publish job: `gh release create "$GITHUB_REF_NAME" --latest=false --title "YAML data $GITHUB_REF_NAME" --notes-file ...` with all staged assets attached — release notes are passed inline via `--notes`, not `--notes-file`, to avoid a redundant scratch file; the body is short and machine-generated.
