# Design: Rust Orchestrator Feature Parity

## Context

The Python `OrchestratorCore` in `ClassicLib/ScanLog/OrchestratorCore.py` provides comprehensive crash log analysis with multiple integrated scanners, analyzers, and report generators. The Rust `OrchestratorCore` in `classic-scanlog-core` implements only a subset of these features.

### Existing Architecture

```
┌─────────────────────────────────────────────────────────────────────┐
│                         HybridOrchestrator                          │
│                 (ClassicLib/ScanLog/HybridOrchestrator.py)          │
├─────────────────────────────────────────────────────────────────────┤
│  Single-log processing:                                             │
│    └── Python OrchestratorCore (full features)                      │
│                                                                     │
│  Batch processing (>5 logs):                                        │
│    └── ClassicOrchestrator (ClassicLib/rust/orchestrator_api.py)    │
│         └── Rust Orchestrator (classic_scanlog.Orchestrator)        │
│              └── OrchestratorCore (classic-scanlog-core)            │
└─────────────────────────────────────────────────────────────────────┘
```

### Current State

**Python OrchestratorCore** provides:
- Complete crash log parsing and segmentation
- Report generation (header, error section, footer)
- Plugin analysis with load order support
- FormID extraction and async database lookup
- Suspect scanning (main error + call stack)
- Settings validation (achievements, memory management, archive limit, looksmenu)
- FCX mode handling
- Record scanning (named records)
- Mod detection (conflict, frequent, solutions, important, OPC2)
- FOLON-specific mod detection
- Batch processing with error handling
- Report writing to files

**Rust OrchestratorCore** provides:
- Basic crash log parsing and segmentation
- Simple report generation
- Plugin analysis (basic)
- FormID extraction (no database lookup)
- Suspect scanning (main error + call stack)
- Mod detection (conflict, frequent, solutions, important, OPC2)
- Bounded parallel batch processing

**HybridOrchestrator** currently:
- Always uses Python for `process_crash_log()` (single-log)
- Uses Rust via `ClassicOrchestrator` for `process_crash_logs_batch()` when >5 logs
- Converts Rust `AnalysisResult` to Python tuple format in `_convert_rust_results()`
- Falls back to Python if Rust processing fails

### Stakeholders
- End users expecting consistent analysis results
- Developers maintaining both implementations
- Performance-sensitive batch processing workflows

## Goals / Non-Goals

### Goals
- Achieve functional parity between Rust and Python orchestrators
- Maintain the three-layer Rust architecture (core/py separation)
- **Refactor HybridOrchestrator** to use Rust for both single-log and batch processing
- Enable true drop-in replacement for Python orchestrator
- Preserve or improve performance characteristics

### Non-Goals
- Replacing all Python scanners with Rust (existing hybrid approach is acceptable)
- Backwards compatibility with pre-parity Rust orchestrator API
- Removing Python OrchestratorCore entirely (keep as fallback)

### Critical Constraint: Identical Output

**Report output MUST be identical between Rust and Python implementations.** Users should NOT be able to determine whether a report was generated by Rust or Python. This includes:
- Exact text content and wording
- Formatting (whitespace, line breaks, separators)
- Section ordering and headers
- Emoji and Unicode characters
- Statistics and counters

This is a hard requirement, not a nice-to-have. Any formatting differences are unacceptable.

### Target Architecture

```
┌─────────────────────────────────────────────────────────────────────┐
│                    HybridOrchestrator (Refactored)                  │
│                 (ClassicLib/ScanLog/HybridOrchestrator.py)          │
├─────────────────────────────────────────────────────────────────────┤
│  Single-log processing:                                             │
│    └── Rust OrchestratorCore (feature-complete) ← PRIMARY           │
│    └── Python OrchestratorCore (fallback only)                      │
│                                                                     │
│  Batch processing:                                                  │
│    └── Rust OrchestratorCore (unbounded parallelism) ← PRIMARY      │
│    └── Python OrchestratorCore (fallback only)                      │
└─────────────────────────────────────────────────────────────────────┘
```

## Decisions

### Decision 1: Phased Implementation Approach

**What**: Implement features in 4 phases, prioritizing core pipeline first.

**Why**:
- Allows incremental testing and validation
- Minimizes risk of introducing regressions
- Enables partial deployment if needed

**Alternatives considered**:
- Big-bang implementation: Higher risk, harder to debug
- Feature-by-feature: Too granular, overhead of many PRs

### Decision 2: Database Pool Integration

**What**: Use `classic-database-core` for async SQLite connection pooling.

**Why**:
- Reuses existing infrastructure
- Avoids code duplication
- Maintains consistency with Python's `AsyncDatabasePool`

**Implementation**:
```rust
// In OrchestratorCore
pub struct OrchestratorCore {
    config: AnalysisConfig,
    file_io: FileIOCore,
    parser: LogParser,
    plugin_analyzer: Option<PluginAnalyzer>,
    formid_analyzer: FormIDAnalyzerCore,
    suspect_scanner: Option<SuspectScanner>,
    settings_validator: Option<SettingsValidator>,  // NEW
    record_scanner: Option<RecordScanner>,          // NEW
    db_pool: Option<DatabasePool>,                  // NEW
}
```

### Decision 3: Report Fragment Composition

**What**: Use the existing `ReportFragment` pattern for all report generation.

**Why**:
- Consistent with Python's fragment-based approach
- Enables lazy evaluation and composition
- Immutable by design (thread-safe)

### Decision 4: Statistics Tracking via AnalysisResult

**What**: Extend `AnalysisResult` to include all Python statistics fields.

**Why**:
- Maintains API compatibility with Python return types
- Enables accurate performance comparison

**Implementation**:
```rust
pub struct AnalysisResult {
    // Existing fields...

    // NEW: Python-compatible statistics
    pub scanned: u32,
    pub incomplete: u32,
    pub failed: u32,
    pub trigger_scan_failed: bool,
}
```

### Decision 5: FCX Mode as Configuration Flag

**What**: FCX mode is a configuration flag, not a separate handler class.

**Why**:
- Simpler implementation in Rust
- FCX mode is read-only (no complex state management needed)
- Matches the functional nature of Rust

## Risks / Trade-offs

### Risk 1: Database Pool Complexity
- **Risk**: Async database connections in Rust require careful lifetime management
- **Mitigation**: Use `deadpool-sqlite` or similar battle-tested pool implementation
- **Alternative**: Fall back to sync database access with thread pool

### Risk 2: Report Format Differences
- **Risk**: Any differences in report formatting are unacceptable (see Critical Constraint)
- **Mitigation**: Add byte-for-byte comparison tests between Python and Rust output
- **Mitigation**: Use Python implementation as the canonical reference for all text strings
- **Mitigation**: Extract all report strings to shared constants/templates
- **No Alternative**: Format differences are NOT acceptable - must be fixed before release

### Risk 3: Performance Regression for Small Batches
- **Risk**: Added features may slow down single-log processing
- **Mitigation**: Profile each phase, optimize hot paths
- **Alternative**: Make advanced features optional (lazy initialization)

## Migration Plan

### Phase 1: Core Pipeline (Week 1-2)
1. Add `ReportGenerator` struct for header/footer
2. Implement version checking
3. Add crash data reformatting (simplify logs)
4. Add incomplete/failed detection

### Phase 2: Analysis Integration (Week 2-3)
1. Integrate `SettingsValidator` into pipeline
2. Add `RecordScanner` implementation
3. Add plugin suspect scanning
4. Add FCX mode flag and messages

### Phase 3: Advanced Features (Week 3-4)
1. Add database pool integration
2. Implement async FormID lookup
3. Add loadorder.txt support
4. Add FOLON detection

### Phase 4: API Alignment (Week 4-5)
1. Update `AnalysisConfig` fields
2. Update `AnalysisResult` statistics
3. Add context manager methods
4. Add write_reports_batch
5. Create HybridOrchestrator (optional)

### Rollback
- Each phase is independently deployable
- Fallback to Python orchestrator remains available via factory pattern
- No data migrations required

## Open Questions

1. **Should we implement HybridOrchestrator in Python or Rust?**
   - Python: Easier integration with existing async patterns
   - Rust: Maximum performance, but more complex Python interop
   - **Recommendation**: Start with Python, optimize later if needed

2. **How should we handle the packaging.Version comparison in Rust?**
   - Option A: Port Version parsing to Rust
   - Option B: Pass pre-parsed version tuples from Python
   - **Recommendation**: Option A for true independence

3. **Should RecordScanner be a new Rust crate or part of classic-scanlog-core?**
   - RecordScanner is relatively simple, keep in core
   - **Recommendation**: Add to classic-scanlog-core
