## Why

`CLASSIC_Info.version` in `CLASSIC Data/databases/CLASSIC Main.yaml` currently stores a display-decorated value like `CLASSIC v9.1.0`. Every consumer that needs the underlying SemVer (CMake build guard, GUI `QApplication::setApplicationVersion`, update-check logic) has to strip the `CLASSIC ` prefix — and every consumer that wants the display form has to trust that the prefix is there. Storing a raw SemVer in the YAML and computing the decorated string at the display boundary eliminates the strip-or-assume coupling and aligns `CLASSIC_Info.version` with the Cargo/CMake `project(VERSION ...)` fields that already store `9.1.0` directly.

## What Changes

- **BREAKING** (YAML schema contract): `CLASSIC_Info.version` value format changes from `CLASSIC v<MAJOR>.<MINOR>.<PATCH>[-pre]` to bare SemVer `<MAJOR>.<MINOR>.<PATCH>[-pre]` (no `CLASSIC v` prefix). Example: `v9.1.0` or `9.1.0` rather than `CLASSIC v9.1.0`. The YAML ships under `CLASSIC Data/databases/CLASSIC Main.yaml`.
- Update the YAML producers (`set_version.ps1`, `CLASSIC Main.yaml` itself) to write the bare form.
- Update YAML consumers so the display-decorated string (`CLASSIC v<SemVer>`) is produced at the point of use rather than assumed to live in the YAML:
  - `classic-gui/src/main.cpp` — drop the `startsWith("CLASSIC v") ? mid(9)` strip and consume the bare SemVer directly.
  - `classic-cli/CMakeLists.txt` — update the `file(STRINGS ... REGEX ...)` match pattern and capture group so SemVer extraction still works.
  - `classic-scanlog-core` report generator — derive the `AUTOSCAN REPORT GENERATED BY CLASSIC v<SemVer>` string by prepending the `CLASSIC v` decoration at format time rather than relying on the decoration coming from the YAML.
- Update the test fixtures in `business-logic/classic-config-core` (unit + integration), `business-logic/classic-scanlog-core`, and `cpp-bindings/classic-cpp-bridge/src/config_tests.rs` that hardcode `version: CLASSIC v...` to use the bare form.
- Bump `schema_version` in `CLASSIC Main.yaml` from `"1.0"` to `"2.0"` (MAJOR bump — the value shape for an existing key has changed, which per the `yaml-update-delivery` contract is a breaking change).

## Impact

- **Affected YAML**: `CLASSIC Data/databases/CLASSIC Main.yaml` (value + `schema_version` bump).
- **Affected Rust**: `business-logic/classic-scanlog-core/src/report.rs` (report header formatting), test fixtures in `business-logic/classic-config-core/{src/yamldata_tests.rs,src/shippable_tests.rs,src/config_tests.rs,tests/integration_tests.rs}` and `business-logic/classic-scanlog-core/src/{report_tests.rs,orchestrator_tests.rs}`.
- **Affected C++**: `classic-gui/src/main.cpp` (prefix-strip removal), `classic-cli/CMakeLists.txt` (regex + capture group), `cpp-bindings/classic-cpp-bridge/src/{config_tests.rs,markdown_tests.rs,update_tests.rs}` (fixture strings).
- **Affected tooling**: `set_version.ps1` — YAML-write pattern no longer injects `CLASSIC v`; comments/help describing "this will be prefixed with CLASSIC v" must be updated.
- **Affected docs**: `docs/api/classic-config-core-yaml-schema.md` (consumed-keys contract), `docs/api/yaml-update-delivery.md` (MAJOR-bump example gains a real precedent), and any contributor docs that show a `CLASSIC_Info.version:` example.
- **Client compatibility**: Older CLASSIC installs still in the field that pull the newly-published `CLASSIC Main.yaml` will strip `CLASSIC v` from something that no longer has it. Because existing consumers use `startsWith("CLASSIC v") ? mid(9) : as-is` (GUI) or regex with the `CLASSIC ` prefix baked in (CMake, PS), they will either produce a SemVer unchanged or fail to match entirely. See `design.md` for the compatibility-bridging strategy.
- **Binding surfaces**: No public Rust/CXX/Node/Python API shape changes — `ConfigData.classic_version` remains a `String`; only its value-shape contract (now bare SemVer, not `CLASSIC v...`) changes. Binding parity gates (CXX/Node/Python) should stay green.
