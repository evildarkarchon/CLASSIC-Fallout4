## Context

`CLASSIC Data/databases/CLASSIC Main.yaml` currently has `CLASSIC_Info.version: CLASSIC v9.1.0`. The value is a display-decorated string — it mixes the product name, the literal `v`, and the SemVer into one field. Three different callers already pay for that decoration:

- **`classic-cli/CMakeLists.txt`** (lines 84–104) uses `file(STRINGS ... REGEX "^[ \t]+version:[ \t]+CLASSIC[ \t]+v[0-9]+\\.[0-9]+\\.[0-9]+")` followed by a capture-group rewrite that drops `CLASSIC ` + `v` to produce `CLASSIC_APP_VERSION=9.1.0` for the `PROJECT_VERSION STREQUAL` guard.
- **`classic-gui/src/main.cpp`** (lines 178–182) reads the YAML value, then `if (qVersion.startsWith("CLASSIC v")) qVersion = qVersion.mid(9).trimmed();` before handing it to `QApplication::setApplicationVersion`.
- **`set_version.ps1`** (line 314) writes `CLASSIC v<NewVersion>` into the YAML by pattern-matching on the literal prefix.

Meanwhile, one caller **does** want the decoration: the scanlog report header (`classic-scanlog-core/src/report.rs:406–409`) prints `AUTOSCAN REPORT GENERATED BY {classic_version}`, so today it prints `AUTOSCAN REPORT GENERATED BY CLASSIC v9.1.0` only because the decoration is baked into the YAML string that flows all the way down from `ConfigData.classic_version`.

The value flows through: `CLASSIC Main.yaml` → `yaml_ops.get_string_value("CLASSIC_Info.version", "")` → `ConfigData.classic_version` (`yamldata.rs:1642`) → `AnalysisConfig.classic_version` (`orchestrator.rs:393`) → `ReportGenerator::with_config(classic_version, ...)` (`report.rs:384`) → report header. The other consumers (CMake, GUI, PowerShell writer) read the YAML string directly, not through the Rust load path.

The decorated format also collides with `Cargo.toml`'s `version = "9.1.0"` and CMake's `project(VERSION 9.1.0)` — both of which already store bare SemVer. `set_version.ps1` is effectively the only thing keeping them in lockstep, and it does so by special-casing the YAML write pattern.

## Goals / Non-Goals

**Goals:**

- Make `CLASSIC_Info.version` a bare SemVer string (`v9.1.0` form) so every consumer that needs SemVer reads it directly.
- Preserve the existing user-visible output `AUTOSCAN REPORT GENERATED BY CLASSIC v9.1.0` by prepending the `CLASSIC ` decoration inside the report generator at format time.
- Bump `schema_version` to `"2.0"` so the existing `yaml-schema-versioning` gate rejects the new shape on older clients — no silent parsing of a newer file against older assumptions.
- Keep `set_version.ps1 -Version "9.2.0"` a single-argument operation: callers still pass bare SemVer and the script writes it bare everywhere.

**Non-Goals:**

- Changing any public Rust / CXX / Node / Python API shape. `ConfigData.classic_version` stays `String`.
- Changing the `CLASSIC_VERSION` constant in `classic-web-core/src/lib.rs` (`"8.0.0"`) — that is a user-agent constant, not loaded from the YAML, and is out of scope.
- Adding a loader-side normalization that silently strips a legacy `CLASSIC v` prefix. The `schema_version` gate is the right rejection point; a silent strip would mask real schema violations.
- Touching the `classic-tui` hard-coded `"CLASSIC v9.0.0"` title literal (`ui.rs:95`). It's a display-only string that should eventually read from the same YAML, but that is a separate cleanup not gated by this change.
- Renaming the field. The YAML key stays `CLASSIC_Info.version`.

## Decisions

### D-01 Stored format is `v<MAJOR>.<MINOR>.<PATCH>[-pre]` (retain the leading `v`)

- The YAML value will be `v9.1.0`, not `9.1.0`.
- **Alternatives considered:**
  - Bare `9.1.0` (matches Cargo/CMake). Rejected: the spec scenarios already contemplate both forms, and the leading `v` aids humans scanning release tags (which follow `v*` namespace); the difference is cosmetic in Rust code (consumers pass the value through unchanged) but the `v` matters to the scanlog report, which prints `CLASSIC v9.1.0` with the `v`.
  - Store bare SemVer and have the report literal the `v` in its format string. Rejected: splits responsibility in two places — the report prepends the literal product-name, but the leading `v` would also have to be prepended, turning one decoration concern into two.
- **Consequence:** the CMake regex changes from `^[ \t]+version:[ \t]+CLASSIC[ \t]+v([0-9]+\\.[0-9]+\\.[0-9]+)` to `^[ \t]+version:[ \t]+v?([0-9]+\\.[0-9]+\\.[0-9]+)`. The `v?` keeps the match tolerant if a future YAML drops the `v`, but the error message should still expect `v<X.Y.Z>` as the canonical form.
- **Consequence:** `set_version.ps1 -Version "9.2.0"` now writes `version: v9.2.0` (the script prepends a literal `v`, not `CLASSIC v`).

### D-02 `CLASSIC ` decoration is prepended inside `ReportGenerator` at format time

- `ReportGenerator` continues to hold `classic_version: String`, but the value is now the raw YAML value (`v9.1.0`), and the `generate_header` format string becomes `format!("**AUTOSCAN REPORT GENERATED BY CLASSIC {}**\n\n", self.classic_version)`.
- **Alternatives considered:**
  - A `product_display_version(raw: &str) -> String` helper in `classic_shared_core`. Rejected for this change: only one consumer needs it (`scanlog-core`'s report header, plus one similar line at `report.rs:640` for the trailing "Generated by ..." footer). A dedicated helper is future-friendly but is mild over-engineering until a second caller materializes. If the TUI title or another surface later needs it, extract then.
  - Leave the decoration in the YAML and teach callers to strip. Rejected: this is the status quo, and the whole point of this change is to move the coupling out of the YAML contract.
- **Consequence:** every test that asserts the report header matches the old format (e.g., `report_tests.rs`, `orchestrator_tests.rs`, `markdown_tests.rs`) keeps the same expected output literal string; the fixtures feeding the generator must change from `"CLASSIC v9.0.0"` to `"v9.0.0"` (or the current release equivalent).

### D-03 `schema_version` bumps to `"2.0"` in the same commit

- This is a MAJOR bump per `openspec/specs/yaml-schema-versioning/spec.md#Additive schema changes bump MINOR, breaking changes bump MAJOR` — the *shape* of the existing field's value has changed.
- **Alternatives considered:**
  - MINOR bump + loader-side backward-compatibility strip. Rejected: changes the shape of an existing value, which the spec already says is MAJOR. MINOR bumps are for additive, ignore-if-unknown changes.
- **Consequence:** the `classic-config-core` `SchemaCompat { accepted_major, minimum_minor }` constant for `MAIN_YAML` must bump `accepted_major` from `1` to `2`. Older clients that pull this YAML via the `yaml-update-delivery` channel will reject it and fall back to their bundled copy — exactly the designed behavior.
- **Consequence:** every in-tree fixture that ships a `schema_version: "1.0"` block together with a `version: CLASSIC v...` string must move to `schema_version: "2.0"` + `version: v...` simultaneously. This matters for shippable-loader tests that previously validated the 1.0 contract; some of those tests should stay (they test the 1.0 gate behavior) but the fixtures that also set `version:` must not conflate the two schema eras.

### D-04 `classic-gui/src/main.cpp` drops the prefix-strip and consumes the value directly

- Remove the `startsWith("CLASSIC v")` branch at lines 178–182. Replace with a bare SemVer pass-through: `QString qVersion = classic::toQString(version).trimmed();` and then `app.setApplicationVersion(qVersion);`. Qt accepts the `v9.1.0` form as an application-version string (it is opaque to Qt).
- **Alternatives considered:**
  - Strip a single leading `v` for Qt's conventional display. Rejected for minimality — it's not required for correctness, and `QApplication::applicationVersion()` is already used opaquely elsewhere. If a display site later wants the bare form, it can strip locally.
- **Consequence:** the error message example text on line 187 that mentions `"CLASSIC v9.0.0"` updates to `"v9.0.0"` or `"9.0.0"` to stay consistent with what the YAML now holds.

### D-05 Ordering: YAML bump and every regex/consumer must land in one commit

- The CMake `file(STRINGS ... REGEX ...)` failure mode is a configure-time FATAL_ERROR. If the YAML bumps to `v9.1.0` but the CMake regex still requires the `CLASSIC ` literal, the next CLI build fails immediately.
- The PowerShell `set_version.ps1` write pattern is the same: a YAML that bumped to `v9.1.0` but a script still expecting `CLASSIC v` will produce a no-op-or-corrupt write on the next `set_version.ps1 -Version X.Y.Z` run.
- **Consequence:** tasks.md enforces single-commit landing for: YAML value + schema_version bump + CMake regex + PS1 write pattern + GUI strip removal + scanlog report format change + `SchemaCompat.accepted_major` bump.

### D-06 Binding surfaces need no code changes — only fixture updates

- CXX bridge exposes `yaml_data_classic_version(&YamlData) -> &str` (`config.rs:70`), Node exposes its own wrapper, Python exposes `.classic_version` on the PyO3 config object. All three return whatever string is in `ConfigData.classic_version`. Changing the string value does not change the binding shape, so the CXX / Node / Python parity gates should not flag drift.
- **Alternatives considered:**
  - Add a companion binding accessor for the "display" form (the `CLASSIC v<SemVer>` variant). Rejected for this change: no binding caller needs the display form today — the only display-form caller is the Rust-internal scanlog report generator. Add later if a binding surfaces a real need.
- **Consequence:** parity gate runs (`python tools/cxx_api_parity/check_parity_gate.py`, `bun run parity:gate`, the python parity tool) are *verification* steps in tasks.md rather than content-producing steps.

## Risks / Trade-offs

- **Risk:** Someone lands the YAML bump without updating the CMake regex in the same commit → CLI build fails at configure time with the existing `FATAL_ERROR`. → **Mitigation:** single-commit ordering (D-05), plus explicit tasks.md step ordering: update CMake regex *before* testing the YAML change end-to-end.
- **Risk:** Someone lands the YAML bump without updating `set_version.ps1` → the next version bump silently produces no-op or malformed writes. → **Mitigation:** single-commit ordering (D-05) plus a post-change `./set_version.ps1 -Version 9.1.0 -WhatIf` smoke-test in tasks.md to confirm the pattern still matches.
- **Risk:** A scan-log fixture or snapshot that hardcodes the old `"CLASSIC v9.0.0"` decorated string slips past `cargo test` (because the decoration is *reintroduced* by the report generator, the *final* report text is unchanged — so fixtures that assert on the *final* header text still pass, but fixtures that assert on the *intermediate* `ConfigData.classic_version` value or the bridge accessor fail). → **Mitigation:** an explicit tasks.md step to grep all test files for the old decorated literal and disambiguate each occurrence (final-report text stays, intermediate-string fixtures get updated).
- **Risk:** An older installed CLASSIC binary still in the field downloads the new YAML via the yaml-update-delivery channel → the `schema_version: "2.0"` gate rejects it, bundled file wins. This is the intended behavior, but it means older users do not get data-only YAML updates shipped alongside this schema bump until they also upgrade their binary. → **Mitigation:** coordinate this change with a CLASSIC binary release bump, not an isolated YAML-only publish.
- **Trade-off:** Some readability is lost — a casual reader scanning `CLASSIC Main.yaml` sees `version: v9.1.0` without the product-name context. The file header comment line 1 (`# DYNAMIC DATABASE for Crash Log Auto Scanner & Setup Integrity Checker (CLASSIC) v7.0+`) already identifies the file as CLASSIC's, so the loss is minor. → Accepted.

## Migration Plan

1. Land this change as part of the `classic-9.1` / `classic-next` line, co-released with a CLASSIC binary version bump (existing `v9.1.0-milestone` work).
2. **No user migration step needed.** Users never hand-edit `CLASSIC_Info.version`; it is overwritten on every `set_version.ps1` run and redistributed via `yaml-update-delivery`.
3. **Rollback strategy:** If a regression is found before release, revert the single commit. Because the YAML, regex sites, schema bump, and report-generator format change all land together, a single-commit revert leaves no half-migrated state. Backups written by `set_version.ps1` into `version_backups/` provide a second safety net for the YAML file specifically.
4. **Post-merge verification:** run the canonical local CI sequence from `.agents/skills/classic-project-guide/references/repo-guide.md`: `cargo test --workspace`, `./rebuild_rust.ps1 -Target python` + python parity pytest, `classic-cli/build_cli.ps1 -Test`, `classic-gui/build_gui.ps1 -Test`, CXX/Node/Python parity gates (expected: all green; no binding drift).
