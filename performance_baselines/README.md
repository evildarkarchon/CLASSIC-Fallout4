# Performance Baselines

This directory contains performance baseline data generated by the async pipeline tests.

## Rust DB Baseline Policy

Rust DB benchmark baselines are local-only artifacts.

- Criterion baseline directories are stored under `ClassicLib-rs/target/criterion/` (gitignored).
- Do not commit raw Rust DB baseline captures by default.
- If a PR needs shareable comparison output, export a focused JSON report and include it only when explicitly requested.

## Phase 5 Criterion Baseline Workflow

Phase 5 proof for `classic-scanlog-core` stays in the existing `scanlog_benchmarks` harness and uses local Criterion baselines.

```powershell
# Save a local before baseline for the Phase 5 groups only
$env:BENCH_MODE = "thorough"
cargo bench -p classic-scanlog-core --manifest-path ClassicLib-rs/Cargo.toml --bench scanlog_benchmarks phase5_ -- --save-baseline phase5-before

# Compare the current Phase 5 groups against that saved baseline
cargo bench -p classic-scanlog-core --manifest-path ClassicLib-rs/Cargo.toml --bench scanlog_benchmarks phase5_ -- --baseline phase5-before
```

- Keep raw Criterion output under `ClassicLib-rs/target/criterion/` local-only by default.
- Do not commit the raw `phase5-before` baseline directory unless a later request explicitly asks for a shareable export.
- Phase 5 hotspots to compare live in `scanlog_benchmarks` under the `phase5_cached_regex_paths`, `phase5_detect_mods_important`, and `phase5_bridge_crash_pattern_replica` groups.
- The committed proof artifact for this workflow lives at `.planning/phases/05-pattern-caching-and-performance/05-BENCHMARK-PROOF.md`.
- Treat a stable comparison that clears the repo's existing `warning > 5%` threshold as the minimum meaningful win bar for this phase.
- If a hotspot does not clear that bar, the Phase 5 summary must explain why the structural change still shipped per D-12 instead of claiming benchmark proof that is not there.

See `docs/performance/rust_db_benchmark_baseline.md` for:

- canonical DB/FormID scenario IDs,
- command set for capture and comparison,
- per-scenario absolute/relative delta schema,
- threshold policy (`warning > 5%`, `fail > 10%`),
- reproducibility and baseline refresh rules.

## Files

- `async_pipeline_baseline_latest.json` - The most recent baseline data (overwritten on each test run)
- `async_pipeline_baseline_YYYYMMDD_HHMMSS.json` - Timestamped baseline data for historical tracking

## Baseline Data Structure

Each JSON file contains the following performance metrics:

```json
{
  "test_date": "2023-XX-XX XX:XX:XX",
  "log_count": 30,
  "total_time": 1.2345,
  "throughput": 24.3,
  "pipeline_efficiency": 0.998,
  "stage_breakdown": {
    "reformat_percent": 46.9,
    "load_percent": 23.9,
    "process_percent": 17.2,
    "write_percent": 11.8
  }
}
```

## Usage

### Running Performance Tests

```bash
# Run the comprehensive baseline test
python -m pytest tests/test_async_pipeline.py::TestAsyncPerformanceBaselines::test_comprehensive_pipeline_baseline -s
```

### Comparing Performance

You can use these baseline files to:
- Track performance improvements over time
- Detect performance regressions
- Compare performance across different system configurations
- Identify which pipeline stages need optimization

## Metrics Explanation

- **total_time**: Total time to process all crash logs (seconds)
- **throughput**: Number of logs processed per second
- **pipeline_efficiency**: Ratio of productive time vs. total time (closer to 1.0 is better)
- **stage_breakdown**: Percentage of time spent in each pipeline stage
  - **reformat_percent**: Time spent reformatting logs
  - **load_percent**: Time spent loading files
  - **process_percent**: Time spent processing content
  - **write_percent**: Time spent writing reports

## Performance Targets

Based on baseline testing:
- **Target throughput**: >40 logs/sec
- **Target efficiency**: >95%
- **Target processing time**: <0.025s per log

## Notes

- Performance data is generated during test runs with realistic but mocked operations
- Actual performance may vary based on system specifications and crash log complexity
- This data is primarily for tracking performance trends and detecting regressions
