## Issue 1 — Medium: Silent fallback discards pool error + sets db_pool regardless

**File:** `ClassicLib-rs/business-logic/classic-scanlog-core/src/orchestrator.rs`
**Lines:** 624–647 (`with_database_pool`), 658–679 (`attach_database_pool`)

### Problem

When `FormIDAnalyzerCore::new(Some(pool))` fails, the error is silently swallowed via `.or_else(|_| ...)`.
Additionally, even after the fallback (pool-less analyzer), `self.db_pool = Some(pool)` is still set unconditionally (lines 645, 678). This means `has_database_pool()` returns `true` while the `FormIDAnalyzerCore` is actually running without a pool — misleading state.

### Fix

1. In the `or_else` closure, capture the error and emit `log::warn!()` before falling back.
2. Only set `self.db_pool = Some(pool)` if the primary call (with pool) succeeded — skip it in the fallback path.

**Approach for `with_database_pool`:**
```rust
pub fn with_database_pool(mut self, pool: Arc<DatabasePool>) -> Result<Self> {
    match FormIDAnalyzerCore::new(Some(pool.clone()), ...) {
        Ok(analyzer) => {
            self.formid_analyzer = analyzer;
            self.db_pool = Some(pool);
        }
        Err(e) => {
            log::warn!(
                "FormIDAnalyzerCore failed with database pool ({}), falling back to pool-less mode",
                e
            );
            self.formid_analyzer = FormIDAnalyzerCore::new(None, ...)?;
            // db_pool intentionally NOT set — pool is unavailable
        }
    }
    Ok(self)
}
```

Apply the same pattern to `attach_database_pool`.

## Issue 2 — Low: Two `reqwest::Error` variants, distinction undocumented

**File:** `ClassicLib-rs/business-logic/classic-update-core/src/error.rs`
**Lines:** 9–14

### Problem

`ClientBuild(#[from] reqwest::Error)` and `HttpError(reqwest::Error)` both wrap the same type. The distinction is intentional but not documented, making it easy for future contributors to use the wrong variant.

### Fix

Add inline doc comments explaining the semantic boundary:

```rust
/// Failed to create HTTP client (e.g., invalid TLS config).
/// Uses `#[from]` so `reqwest::Error` converts automatically via `?`.
#[error("Failed to create HTTP client: {0}")]
ClientBuild(#[from] reqwest::Error),

/// HTTP-layer error during a request or response (e.g., network failure, bad status).
/// Mapped explicitly via `.map_err(UpdateError::HttpError)` — NOT auto-converted,
/// because `#[from]` is already taken by `ClientBuild` for the same source type.
#[error("HTTP error: {0}")]
HttpError(reqwest::Error),
```

## Issue 3 — Low: Expired cache branch undocumented in `pool_sqlx.rs`

**File:** `ClassicLib-rs/business-logic/classic-database-core/src/pool_sqlx.rs`
**Lines:** 299–306 (`get_entry`), ~440 (`get_entries_batch`)

### Problem

In `get_entry`, an expired cache entry falls through to the cache-miss stat counters (correct behavior), but there's no comment explaining this. A reader may assume expired entries are entirely skipped from stats.

In `get_entries_batch`, the expired entry branch is similarly undocumented.

### Fix

Add a brief comment at the expired-entry branch:

```rust
} else {
    // Expired entry: evict and treat as a cache miss (counted below).
    self.query_cache.remove(&cache_key);
    debug!("Cache expired for FormID: {} Plugin: {}", formid, plugin);
}
// Falls through to cache-miss counters (covers both cold-miss and expired-miss).
self.stat_cache_misses.fetch_add(1, Ordering::Relaxed);
self.stat_total_queries.fetch_add(1, Ordering::Relaxed);
```

Apply same comment pattern to the expired branch in `get_entries_batch`.

## Issue 4 — Low: `DashMap` shard locking for `f64` mutation undocumented

**File:** `ClassicLib-rs/python-bindings/classic-pybridge-py/src/metrics.rs`
**Lines:** 86–96 (`record_bridge_operation`)

### Problem

`METRICS.entry(operation).or_default()` returns a `RefMut` that holds a `DashMap` shard lock for its lifetime. All field mutations (`count`, `success`, `failure`, `total_time`) happen under that lock — making them collectively atomic per call. This differs from the `Arc<AtomicU64>` approach used in `pool_sqlx.rs`, and is not obvious to a reader.

### Fix

Add a comment explaining the locking:

```rust
pub fn record_bridge_operation(operation: BridgeOperation, duration_secs: f64, success: bool) {
    // `entry().or_default()` returns a `RefMut` that holds the DashMap shard lock
    // for its entire lifetime — all field updates below are collectively atomic per call.
    let mut stats = METRICS.entry(operation).or_default();
    ...
}
```
