# Plan: Move TOML Checks to YAML Crashgen Registry

## Goal
Migrate TOML settings checks so both criteria and user-facing messages come from `Crashgen_Registry` YAML data, with one shared evaluation path used by:
- `classic-scanlog-core` (crash-log settings section checks)
- `classic-scangame-core` (direct `Buffout4/config.toml` checker)

Scope is confirmed: both paths now.

## Recommended Approach
1. Extend `Crashgen_Registry` schema to define full TOML/settings rules (conditions + expected values + messages), while keeping legacy fields for compatibility.
2. Introduce one shared Rust rule evaluator crate/module consumed by both scanlog and scangame.
3. Refactor both current hardcoded implementations to adapter layers that build context and map shared outcomes to each path's output format.
4. Migrate FO4 YAML entries to the new schema and add parity-focused tests.

## YAML Schema Design
Add new optional fields under each `Crashgen_Registry.<name>` entry in `CLASSIC Data/databases/CLASSIC Fallout4.yaml`:

- `settings_rules_version: 1`
- `settings_rules:`
  - `preflight:` list of rules for skip/incompatibility notices (example: Addictol-only skip; Addictol+Buffout incompatibility)
  - `checks:` list of setting rules (example: Achievements, MemoryManager, X-Cell-related keys, ArchiveLimit, F4EE, Baka ScrapHeap special case)

Each check rule should include:
- `id`
- `when` predicate tree (`all` / `any` / `not`, plugin presence, layout/version facts)
- `target` (`section`, `key`, `type`)
- `expect` (expected value/comparator)
- `messages` (`fail`, `fix`, optional `pass`)
- `severity`

Compatibility policy:
- Keep existing `display_section`, `ignore_keys`, `checks` parsed and usable during migration.
- If `settings_rules` is missing or malformed, fall back to legacy behavior for that entry.

## Implementation Steps
1. **Extend raw YAML parsing and models**
   - Update `ClassicLib-rs/business-logic/classic-config-core/src/yamldata.rs`:
     - extend `CrashgenEntryRaw` with parsed rule fields
     - parse `settings_rules_version` and `settings_rules` (permissive skip-on-malformed behavior, with debug logs)
   - Update exports in `ClassicLib-rs/business-logic/classic-config-core/src/lib.rs` if needed.
   - Update parser tests in `ClassicLib-rs/business-logic/classic-config-core/src/yamldata.rs`.

2. **Create shared evaluator (single source of truth)**
   - Add new crate in `ClassicLib-rs/business-logic/` (for example `classic-crashgen-settings-core`) and register it in `ClassicLib-rs/Cargo.toml`.
   - Define:
     - rule domain types (predicates, targets, expected values, message templates)
     - normalized evaluation context (installed plugins, setting map, crashgen/display names, config layout, optional version facts)
     - outcome model (severity, message/fix text, optional structured issue payload, skip action)
   - Add focused unit tests for predicate and action semantics.

3. **Refactor scanlog path to shared evaluator**
   - Update `ClassicLib-rs/business-logic/classic-scanlog-core/src/crashgen_registry.rs` to carry compiled/new rules in `CrashgenEntry`.
   - Update registry build path in `ClassicLib-rs/business-logic/classic-scanlog-core/src/orchestrator.rs` (`build_crashgen_registry`).
   - Refactor `ClassicLib-rs/business-logic/classic-scanlog-core/src/settings_validator.rs`:
     - replace hardcoded check logic/messages with evaluator invocation
     - keep formatting compatible with existing report style
   - Simplify Addictol branching in `ClassicLib-rs/business-logic/classic-scanlog-core/src/orchestrator.rs` so preflight behavior comes from rules, not hardcoded branches.

4. **Refactor scangame TOML checker to shared evaluator**
   - Refactor `ClassicLib-rs/business-logic/classic-scangame-core/src/toml.rs`:
     - remove hardcoded `get_settings_to_check`/special-case branching
     - build evaluation context from parsed TOML + installed DLLs + config layout
     - map evaluator outcomes to `TomlConfigIssue` + message text
   - Update `ClassicLib-rs/business-logic/classic-scangame-core/src/crashgen_orchestrator.rs`:
     - add registry-aware entry point (and keep existing `check(...)` as compatibility wrapper)
   - Update `ClassicLib-rs/business-logic/classic-scangame-core/src/orchestrator.rs` so game-scan crashgen checks can use rule-driven behavior.

5. **Update bindings and adapters impacted by model/API changes**
   - Struct/model propagation for extended `CrashgenEntryRaw`:
     - `ClassicLib-rs/python-bindings/classic-scanlog-py/src/orchestrator.rs`
     - `ClassicLib-rs/python-bindings/classic-config-py/src/lib.rs`
     - `ClassicLib-rs/business-logic/classic-scanlog-core/src/orchestrator.rs` tests using `CrashgenEntryRaw` literals
   - If scangame config API is expanded to accept rule/registry data, update:
     - `ClassicLib-rs/node-bindings/classic-node/src/scangame.rs`
     - `ClassicLib-rs/python-bindings/classic-scangame-py/src/orchestrator.rs`
     - `ClassicLib-rs/python-bindings/classic-scangame-py/src/crashgen_orchestrator.rs`

6. **Migrate FO4 YAML data**
   - Populate `Crashgen_Registry` rule blocks in `CLASSIC Data/databases/CLASSIC Fallout4.yaml` for Buffout/Addictol/default.
   - Move existing hardcoded criteria and message text into these YAML rules.

7. **Docs/spec synchronization**
   - Update `openspec/specs/crashgen-schema-registry/spec.md` to reflect that criteria and messages are now YAML-driven (not only named-check assignment).
