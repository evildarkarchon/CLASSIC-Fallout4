# Crashgen Registry Rules Wiring Plan (Active Bindings)

## Scope and decision
- Scope: active bindings only (`ClassicLib-rs/python-bindings/*`, `ClassicLib-rs/node-bindings/classic-node/*`), no `deprecated/` updates.
- Compatibility: breaking API updates are allowed.

## Implementation plan
1. Python config + scanlog rule plumbing
- Update `ClassicLib-rs/python-bindings/classic-config-py/src/lib.rs` and `ClassicLib-rs/python-bindings/classic-config-py/classic_config.pyi` so `YamlData.crashgen_registry` exposes full `settings_rules` payload (not just `has_settings_rules`).
- Update `ClassicLib-rs/python-bindings/classic-scanlog-py/src/orchestrator.rs` to parse `settings_rules` from Python input and set `CrashgenEntryRaw.settings_rules` instead of hardcoded `None`.
- Update `ClassicLib-rs/python-bindings/classic-scanlog-py/src/settings_validator.rs` and `ClassicLib-rs/python-bindings/classic-scanlog-py/classic_scanlog.pyi` to accept/build a rules-aware crashgen entry and expose the modern full-settings scan path.

2. Python scangame rule plumbing (including missed active surface)
- Update `ClassicLib-rs/python-bindings/classic-scangame-py/src/orchestrator.rs` and `ClassicLib-rs/python-bindings/classic-scangame-py/classic_scangame.pyi` so `GameScanConfig` accepts `crashgen_settings_rules` and forwards it to core config.
- Update `ClassicLib-rs/python-bindings/classic-scangame-py/src/crashgen_orchestrator.rs` to use rules-aware check APIs (`check_with_rules`) and accept rules parameters in exposed methods/functions.
- Update `ClassicLib-rs/python-bindings/classic-scangame-py/src/toml_check.rs` so `CrashgenChecker`/`check_crashgen_config` can pass rules (`new_with_rules`) instead of ruleless-only construction.

3. Node config/scanlog/scangame rule plumbing
- Update `ClassicLib-rs/node-bindings/classic-node/src/config.rs` to expose `YamlData.crashgenRegistry` with per-entry `displaySection`, `ignoreKeys`, `checks`, `settingsRulesVersion`, and `settingsRules`.
- Update `ClassicLib-rs/node-bindings/classic-node/src/scanlog.rs`:
  - extend `JsAnalysisConfig` with crashgen registry/rules data,
  - map it in `_js_config_to_core`,
  - include it in `create_analysis_config_from_yaml_content` output so YAML-built config round-trips through `process_log`/batch flows.
- Update `ClassicLib-rs/node-bindings/classic-node/src/scangame.rs`:
  - extend `JsGameScanConfig` with `crashgenSettingsRules`,
  - map to `GameScanConfig.crashgen_settings_rules`,
  - update crashgen checker/orchestrator exported helpers to support rules-aware paths.

4. Type surface + tests/parity updates
- Regenerate and commit `ClassicLib-rs/node-bindings/classic-node/index.d.ts`.
- Update Node tests to assert rules are available and consumed:
  - `ClassicLib-rs/node-bindings/classic-node/__test__/config.spec.ts`
  - `ClassicLib-rs/node-bindings/classic-node/__test__/scanlog.spec.ts`
  - `ClassicLib-rs/node-bindings/classic-node/__test__/scangame.spec.ts`
  - `ClassicLib-rs/node-bindings/classic-node/__test__/parity_tier1.spec.ts`
  - `ClassicLib-rs/node-bindings/classic-node/__test__/fixtures/tier1_parity.fixtures.ts`
  - `ClassicLib-rs/node-bindings/classic-node/__test__/fixtures/tier1_regression.fixtures.ts` (if required by signature changes)
- Update Python `.pyi` stubs for any changed constructors/functions so runtime and typing stay in sync.

## Verification
1. Rust workspace quality gates
- `cargo fmt --all --manifest-path ClassicLib-rs/Cargo.toml -- --check`
- `cargo clippy --workspace --all-targets --all-features --manifest-path ClassicLib-rs/Cargo.toml -- -D warnings`
- `cargo test --workspace --manifest-path ClassicLib-rs/Cargo.toml`

2. Targeted binding validation
- `cargo test -p classic-config-py -p classic-scanlog-py -p classic-scangame-py --manifest-path ClassicLib-rs/Cargo.toml`

3. Node binding/parity gates (from `ClassicLib-rs/node-bindings/classic-node`)
- `bun install`
- `bun run build`
- `bun run parity:gate:local`
- `bun run test:bun`
- `bun run test:node`

4. End-to-end behavior checks
- Confirm Python and Node can both:
  - read crashgen registry entries with `settings_rules` from YAML content path,
  - pass rules into scanlog/scangame check paths,
  - produce rule-driven crashgen settings findings matching core behavior.

## Migration notes (breaking)
- Python signatures for settings validator and scangame crashgen checks will change to rules-aware forms.
- Node `JsAnalysisConfig` and `JsGameScanConfig` shapes will change; TS consumers must update object literals to include new rule fields where required.
