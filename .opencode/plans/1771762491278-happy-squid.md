# Fix Code Review Issues

## Background

Three issues were identified in a recent code review. This plan addresses them in priority order.

---

## Issue 1 (Medium) — `orchestrator.rs`: `Result` return type contract mismatch

### Problem
`with_database_pool` and `attach_database_pool` both return `Result<Self>` / `Result<()>`, but pool installation failure silently returns `Ok(...)` with the pool discarded. Callers cannot distinguish "pool attached" from "pool silently dropped."

The secondary fallback (`FormIDAnalyzerCore::new(None, ...)`) is also dead code today — `FormIDAnalyzerCore::new` is **documented as infallible** (always returns `Ok(_)`), so the `?` on the fallback never fires.

### Fix
Remove the silent fallback. Let the primary `FormIDAnalyzerCore::new(Some(pool), ...)` error propagate directly via `?`. Since `FormIDAnalyzerCore::new` is currently infallible, this doesn't change runtime behavior — it only corrects the contract.

### File
`ClassicLib-rs/business-logic/classic-scanlog-core/src/orchestrator.rs`

#### `with_database_pool` (lines 624–658) — replace with:
```rust
pub fn with_database_pool(mut self, pool: Arc<DatabasePool>) -> Result<Self> {
    self.formid_analyzer = FormIDAnalyzerCore::new(
        Some(pool.clone()),
        self.config.show_formid_values,
        self.config.crashgen_name.clone(),
        self.config.mods_core.clone(),
        self.config.mods_freq.clone(),
        self.config.mods_conf.clone(),
    )?;
    self.db_pool = Some(pool);
    Ok(self)
}
```

#### `attach_database_pool` (lines 669–699) — replace with:
```rust
pub fn attach_database_pool(&mut self, pool: Arc<DatabasePool>) -> Result<()> {
    self.formid_analyzer = FormIDAnalyzerCore::new(
        Some(pool.clone()),
        self.config.show_formid_values,
        self.config.crashgen_name.clone(),
        self.config.mods_core.clone(),
        self.config.mods_freq.clone(),
        self.config.mods_conf.clone(),
    )?;
    self.db_pool = Some(pool);
    Ok(())
}
```

**Callers** (C++ bridge `scanner.rs:87`, PyO3 `orchestrator.rs:923`) already handle `Result` via `.map_err(...)? ` — no changes needed there.

## Issue 2 (Low) — `error.rs`/`github.rs`: Incorrect doc comments claim `JsonError` can be returned

### Problem
`JsonError(#[from] serde_json::Error)` exists in `UpdateError`, but `github.rs` methods never produce it — `reqwest`'s `.json()` returns `reqwest::Error`, which maps to `HttpError`, not `JsonError`. The doc comments on `get_latest_release` and `get_all_releases` incorrectly list `JsonError` as a possible error.

The `JsonError` variant itself is kept (it's valid public API surface for callers using `serde_json::from_str` with `?`), but the documentation needs correction.

### Files
- `ClassicLib-rs/business-logic/classic-update-core/src/github.rs` — fix `# Errors` doc sections for `get_latest_release` (line ~236) and `get_all_releases` (line ~294) to remove references to `JsonError`
