# Plan: reject non-installable YAML names before publish

## Goal

Block `client-schema-ranges.yaml` entries whose `files[].name` would be rejected by the client at install time, so CI cannot publish a manifest that later fails `classic-update-core` validation for every user.

## Findings

- `business-logic/classic-update-core/src/yaml_update.rs` already contains the authoritative client-side cache-basename contract in `is_valid_cache_file_name()`, plus `is_reserved_windows_device_basename()` and `windows_normalized_cache_file_key()`.
- That Rust validator already rejects the concrete bad cases from the report, including Windows device names with extensions such as `NUL.yaml` and `CON.txt`, plus ADS/path-like forms such as `foo:bar.yaml`.
- `tools/publish_yaml_data/validate.py::load_shippable_names()` currently enforces Windows-identity duplicate detection and schema-range shape/order checks, but it does not yet reject names that are invalid cache basenames.
- `tools/publish_yaml_data/generate_manifest.py::load_ranges()` mirrors the duplicate/range validation for direct script invocation, so it should gain the same basename gate to avoid a bypass outside the main workflow.
- The publish workflow already runs `validate.py` before `generate_manifest.py`, and the repo guide says changes to YAML publish loader/manifest behavior should be reflected in `docs/api/yaml-update-delivery.md`.

## Implementation

1. Add a shared Python cache-name validator for publish tooling.
- Introduce a small helper module under `tools/publish_yaml_data/` that mirrors the Rust cache-basename rules used by `classic-update-core`.
- Keep the implementation string-based and OS-independent so Linux CI and Windows local runs enforce the same contract without relying on host-specific `Path` parsing.
- Mirror the Rust behavior for: empty / `.` / `..`, `/` and `\\`, `:`, embedded NUL, trailing dot/space, and reserved Windows device basenames (`CON`, `PRN`, `AUX`, `NUL`, `COM1`..`COM9`, `LPT1`..`LPT9`, with or without extensions, case-insensitively).
- Keep the existing Windows normalization helper in the same shared module so duplicate detection and installability checks stay colocated.

2. Enforce the new rule in `validate.py`.
- Call the shared validator from `load_shippable_names()` immediately after extracting `entry["name"]` and before adding it to the returned name set.
- Raise `SystemExit` with a `FAIL:` diagnostic that names the offending entry and states that the file name is not a valid cache/install basename.
- Preserve the current duplicate-entry and schema-range diagnostics unchanged unless the invalid-name check fires first.

3. Mirror the rule in `generate_manifest.py`.
- Reuse the same shared validator in `load_ranges()` so `python tools/publish_yaml_data/generate_manifest.py ...` cannot emit a manifest for names the client would refuse.
- Keep this pass structurally aligned with `validate.py`: basename validity first, then duplicate detection, then schema-range validation.

4. Extend Python regression coverage.
- Expand `tools/publish_yaml_data/tests/test_validate.py` or add a sibling test module that exercises both `load_shippable_names()` and `load_ranges()` against invalid names.
- Cover at least: `NUL.yaml`, `CON.txt`, ADS/path-like forms with `:`, separator/path traversal forms, trailing dot/space, and `.` / `..`.
- Add a positive regression for allowed lookalikes such as `NULL.yaml`, `COM10.yaml`, or `CON_main.yaml` so the publish gate does not over-reject beyond the Rust contract.
- Keep one subprocess-style test for `validate.py` so the CLI still surfaces the failure text and exit code the workflow expects.

5. Update maintainer-facing docs.
- Amend `docs/api/yaml-update-delivery.md` to note that publish validation now rejects not only malformed YAML/schema headers but also any shippable filename that violates the client cache-basename contract.
- Keep the doc update narrow and workflow-focused; no Rust API docs should be needed because the runtime contract itself is unchanged.

## Validation

- `python -m pytest tools/publish_yaml_data/tests/test_validate.py -q`
- `python tools/schema_version_gate.py --repo-root .`
- `python tools/publish_yaml_data/validate.py --databases-dir "CLASSIC Data/databases" --schema-ranges "CLASSIC Data/databases/client-schema-ranges.yaml"`

## Notes

- No Rust code change is required for this fix; the client already enforces the desired behavior.
- Centralizing the Python-side rules in one helper avoids drift between `validate.py` and `generate_manifest.py` the next time the cache-basename contract changes.
