# Plan: close YAML update filename + publish duplicate gaps

## Goal

Fix the two reported hardening gaps without widening behavior beyond the specific Windows-path identity issues that were found.

## Findings

- `business-logic/classic-update-core/src/yaml_update.rs:is_valid_cache_file_name()` already rejects separators, NUL, trailing dot/space, rooted paths, multi-component paths, and Windows DOS device basenames, but it still accepts `:`. That leaves NTFS alternate data stream names like `CLASSIC Main.yaml:alt` valid at the manifest boundary.
- The runtime manifest validator already has a Windows-identity normalization helper, `windows_normalized_cache_file_key()`, which trims trailing spaces/dots and lowercases before duplicate checks.
- `tools/publish_yaml_data/validate.py::load_shippable_names()` currently deduplicates `files[].name` by exact string equality only, so a publish can succeed with names that the client later rejects as duplicates.

## Implementation

1. Tighten the Rust cache-basename validator.
- Update `is_valid_cache_file_name()` so `:` is rejected alongside `/`, `\\`, and embedded NUL.
- Adjust the nearby doc comment to explicitly call out Windows alternate data stream / stream-name forms as invalid.
- Keep the change minimal; do not broaden this patch to unrelated Win32-invalid punctuation unless a new requirement appears.

2. Add Rust regression coverage for the ADS case.
- Extend `business-logic/classic-update-core/src/yaml_update_tests.rs` with direct validator tests such as `CLASSIC Main.yaml:alt` and `foo:bar.yaml`.
- Extend `business-logic/classic-update-core/tests/yaml_update_tests.rs` with a manifest-level regression proving `validate_manifest()` rejects an ADS-style `files[].name`.
- Reuse the existing `ManifestInvalid`/basename-style assertions so the new tests stay aligned with current error wording.

3. Align publish-time duplicate detection with runtime Windows semantics.
- Add a small helper in `tools/publish_yaml_data/validate.py` that mirrors `windows_normalized_cache_file_key()` semantics: trim trailing spaces/dots and compare case-insensitively.
- Use that normalized key for duplicate detection while preserving the original `name` string in diagnostics and in the returned shippable-name set.
- Keep downstream file-presence checks exact/case-sensitive for now so this patch only addresses the self-rejecting-manifest bug.

4. Add publish-tool regression coverage.
- Add a focused pytest module under `tools/publish_yaml_data/tests/` covering raw duplicates, case-only duplicates, and trailing-dot/space variants if they are still parseable inputs.
- Assert that `load_shippable_names()`/the script exits with the duplicate-entry diagnostic for those inputs.

5. Optional same-pass hardening.
- Consider mirroring the same normalized duplicate check inside `tools/publish_yaml_data/generate_manifest.py::load_ranges()`. The official workflow already runs `validate.py` first, so this is not required to fix the reported release-path bug, but it would close the direct-script bypass with a very small follow-up diff.

## Validation

- `cargo fmt --all`
- `cargo test -p classic-update-core yaml_update`
- `python -m pytest tools/publish_yaml_data/tests -q` if the new tool test module is added
- `python tools/publish_yaml_data/validate.py --databases-dir "CLASSIC Data/databases" --schema-ranges "CLASSIC Data/databases/client-schema-ranges.yaml"`

## Docs / Artifacts

- No `docs/api/*` update is expected because this is validator hardening rather than a public API shape change.
- Keep the Rust and Python inline comments/docstrings synchronized with the stricter rules so future duplicate/filename checks continue to mirror runtime behavior.
