# Rust owns Crash Log Scan Run behavior

Crash Log Scan Run behavior lives behind one Rust `scan_run` module interface so CLI, GUI, TUI, Node, Python, and C++ adapters do not preserve the discover-setup-prepare-orchestrator-process-write-move sequence themselves. The module owns Standard/Targeted Crash Log discovery, FCX setup result shaping, Crash Log Scan Intake preparation, adaptive concurrency, analysis, Autoscan Report writing, progress and safe-seam cancellation semantics, failed-log accounting, and Unsolved Logs decisions; adapters remain responsible for typed request projection, event/result mapping, presentation, and user messaging.

The external interface uses one execution entry point with a tagged Standard or Targeted request, a separate opaque cancellation control, and optional non-controlling event observation. Discovery and Rust-selected effective concurrency become available during the run and remain in the terminal result; events preserve execution order, while per-log outcomes always use discovery order. Targeted Crash Log Scan Runs cannot request Unsolved Logs movement, and FCX Mode cannot be requested without Crash Log Scan Setup Context.

Expected lifecycle states and per-log failures are structured result data; run-wide infrastructure failures use stable typed stages. Cancellation stops work only at safe seams, so an admitted Crash Log completes analysis, Autoscan Report persistence, and any Unsolved Logs finalization atomically. `OrchestratorCore`, prepared-run execution, batch lifecycle types, direct report-writing entry points, concurrency helpers, `FcxModeHandler`, `FcxResetError`, and `GLOBAL_FCX_HANDLER` are internal or removed in one coordinated cross-language cutover with no external analysis-only execution path; independently useful parser and analyzer interfaces remain outside this decision.
