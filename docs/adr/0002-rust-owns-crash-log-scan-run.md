# Rust owns Crash Log Scan Run behavior

Crash Log Scan Run behavior lives behind a Rust `scan_run` module so CLI, GUI, TUI, Node, Python, and C++ adapters do not preserve the discover-setup-prepare-orchestrator-process-write-move sequence themselves. The high-level service owns Standard/Targeted Crash Log discovery, FCX setup result shaping, Crash Log Scan Intake preparation, analysis, Autoscan Report writing, progress and cancellation semantics, failed-log accounting, and Unsolved Logs decisions; adapters remain responsible for presentation and user messaging. This concentrates locality for product behavior in Rust, while preserving `OrchestratorCore` as internal analysis implementation behind the scan-run module.

Targeted Crash Log Scan Runs keep the current rule that failed Crash Logs and Autoscan Reports are not moved to Unsolved Logs. Targeted inputs rejected during discovery are returned as run-level discovery data, and Standard no-log cases are returned as run-level result data rather than adapter-specific preflight branches.
