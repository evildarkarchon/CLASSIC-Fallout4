# Use semantic contributions and private Autoscan Report Assembly

Report-relevant analysis is owned by six public Focused Semantic Analyzers: Crashgen Settings, Crash Suspect, Mod Guidance, Plugin Evidence, FormID Finding, and Named Record Finding. Each analyzer accepts owned facts and returns typed semantic findings, counts, states, and authored YAML Data guidance without report lines or presentation mechanics. A private collector coordinates those results for a complete Crash Log Scan Run; a present empty result means analysis completed without a match, absence means the analysis was not performed, and failure is represented only by the typed error path. Private Autoscan Report Assembly alone owns canonical section order, grouping, sorting, headings, separators, padding, icons, markdown, and code-authored prose.

This boundary is a coordinated breaking Rust, CXX, Node, and Python interface cutover. `ReportFragment`, `ReportComposer`, `ReportGenerator`, presentation-only `StringPool` operations, `SettingsValidator`, Python-only `ParallelReportProcessor`, and fragment-producing analyzer entry points are not supported public contracts and have no deprecated aliases or forwarding facades. Independently useful semantic parsing, extraction, validation, matching, and batch utilities remain public. CXX, Node, and Python mechanically project the six owned analyzer inputs and results plus the shared `AnalyzerError` kind, code, and message; parity requires both the positive semantic surface and the absence of the retired presentation surface.

Focused callers receive construction, configuration, malformed-dependency, and operational failures as typed `AnalyzerError` values. During a complete run, failure to construct the reusable analyzer set prevents scheduling and becomes a run-wide `Initialization` infrastructure error. A per-log analyzer execution failure becomes `LogFailureStage::Analysis`, produces no partial Autoscan Report, and does not change the outcomes of other admitted Crash Logs. Empty findings are successful result data and are never used to recover from an error.

For the same Crash Log, YAML Data, scan facts, and options, successful complete runs must persist a byte-identical Autoscan Report. Refactoring collection, analyzer internals, or binding projections must not change section ordering, authored whitespace, Unicode, separators, or the final newline unintentionally. The immutable corpus under `tests/fixtures/autoscan_report_goldens/` enforces this requirement only through the public `scan_run::contract::execute` seam; it does not call private collection or rendering helpers.

This decision preserves the earlier ownership boundaries: [ADR-0001](0001-yaml-crashgen-expectations.md) keeps Crashgen Expectations in YAML Data, [ADR-0002](0002-rust-owns-crash-log-scan-run.md) keeps complete run behavior in Rust, and [ADR-0003](0003-autoscan-report-placement-yaml-data.md) keeps Crashgen Expectation placement in YAML Data while private Rust assembly interprets it.
