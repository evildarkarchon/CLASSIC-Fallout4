## Finding 1: C++ Bridge Parity Gate Tooling

### Problem

No parity gate exists for the CXX bridge surface. The Python and Node gates (`tools/python_api_parity/generate_baseline.py`, `tools/node_api_parity/generate_baseline.py`) parse Rust source files with regex to enumerate exported symbols. The same approach can generate a C++ surface inventory without any new Rust crates.

### Approach: Parse `#[cxx::bridge]` Rust source directly (the only viable approach)

CXX-generated headers are build artifacts written into the Cargo build cache (`target/`), not committed. The single committed header (`include/classic_cxx_bridge/scan_progress_callback.h`) is hand-authored, not generated by CXX. Parsing committed headers is therefore incomplete and fragile.

The Rust `#[cxx::bridge]` mod source blocks are the source of truth and are committed. Every function and type in those mods becomes the C++ API. The `build.rs` explicitly enumerates all 14 bridge source files, so the inventory scope is unambiguous.

**Approach:** Write a Python script at `tools/cpp_api_parity/generate_baseline.py` that:
1. Iterates the 14 source files listed in `ClassicLib-rs/cpp-bindings/classic-cpp-bridge/build.rs` (or hard-codes them matching the build.rs list)
2. For each file, extracts the `#[cxx::bridge(namespace = "...")]` block's `extern "Rust"` items and shared struct/enum declarations
3. Maps them to `namespace::name` identifiers matching what CXX generates in C++
4. Produces a JSON contract baseline comparable to the Python and Node baselines

**Parsing strategy:** Regex extraction is sufficient for this project's CXX bridge shape (the existing Python gates already use regex for Rust `lib.rs` parsing, per `generate_baseline.py`). The bridge mods use consistent patterns: `fn name(...)`, `type Name;`, `struct Name { ... }`, `enum Name { ... }` all inside `mod ffi { ... }` blocks inside `#[cxx::bridge]`.

### Additional option: `syn` for exact AST parsing

If regex extraction proves brittle for complex types (nested generics, CXX-specific attributes), a small Rust CLI tool using `syn` can be invoked by the Python gate script to parse bridge mods as a proper Rust AST and emit JSON.

| Library | Version | Status | Why |
|---------|---------|--------|-----|
| `syn` | 2.0.117 | ALREADY workspace dep (via `cxx-build`) | Parse `#[cxx::bridge]` mod ASTs for accurate type/function extraction if regex is insufficient |

`syn` is already pulled in transitively as a dependency of `cxx-build`. There is no new dep needed if a small Rust CLI tool is added to the workspace. However, a Rust CLI adds build/maintenance overhead. Regex-first is recommended; fall back to `syn`-based CLI only if specific bridge entries are missed by regex.

### New deps needed for CXX parity gate

**No new Rust workspace dependencies required.** The gate script is pure Python (matching the Node and Python gate pattern) and runs against committed Rust source files. `syn` is already present as a transitive dep if a Rust helper is ever needed.

## Finding 2: CXX Header Freshness Gating

### Problem

The Node binding uses committed `index.d.ts` + `check_dts_freshness.py` (rebuild â†’ `git diff` â†’ fail if diff). There is no analogous committed snapshot + diff gate for CXX-generated headers.

### How CXX headers actually work

CXX generates headers into `$OUT_DIR` (inside the Cargo `target/` tree). The `classic-cpp-bridge` crate does NOT commit generated headers â€” only the hand-authored `include/classic_cxx_bridge/scan_progress_callback.h` is committed, and that header is a manually-maintained abstract base class, not generated by CXX.

The actual generated headers (`cxx.h` wrapper, per-bridge include files) live in `target/` and are rebuilt on every compile. C++ consumers include them via the path set by `cxx-build` in its `CXXFLAGS` output.

### What freshness gating means for C++

For C++, "freshness" means: does the committed CXX bridge parity contract (JSON baseline) match what the committed Rust source would generate? This is exactly what the Python gate script (Finding 1) enforces â€” the gate fails if the Rust source diverges from the baseline, which is the signal that the C++ API changed without updating the contract.

The `index.d.ts` pattern (rebuild â†’ diff committed artifact) does NOT apply to CXX-generated headers because those headers are not committed and not the source of truth. **The contract file IS the "committed snapshot" for C++.** The gate script regenerates the surface inventory from Rust source on each CI run and diffs it against the committed baseline JSON.

### Hand-authored headers

`include/classic_cxx_bridge/scan_progress_callback.h` is hand-authored and should be included in the C++ bridge parity baseline as a manually-tracked entry (not regenerated). Changes to it are caught by normal `git diff` in CI.

### New deps needed for CXX header freshness

**No new dependencies.** The committed JSON baseline + Python regeneration gate is the exact equivalent of the `index.d.ts` committed-snapshot pattern, without requiring committed generated headers. No new Rust crates, no new Python packages beyond what the existing parity scripts already use.

## Finding 3: Python `classic_shared` Runtime Helpers

### Current state (already implemented)

`classic-shared-py/src/lib.rs` already implements:
- `RuntimeStats` PyO3 class with `worker_threads: usize` and `is_healthy: bool`
- `get_runtime_stats() -> RuntimeStats` using `get_runtime().metrics().num_workers()`
- `is_runtime_healthy() -> bool` using the same metrics path
- Registered in the `classic_shared` pymodule

The `.pyi` stub at `ClassicLib-rs/foundation/classic-shared-py/classic_shared.pyi` already documents all three (`RuntimeStats`, `get_runtime_stats`, `is_runtime_healthy`).

### What is NOT done

The `classic_shared` module is not tracked in the Python parity gate. `tools/python_api_parity/generate_baseline.py` lists only three Rust target crates (`classic-scanlog-core`, `classic-config-core`, `classic-version-registry-core`) and three Python target modules. `classic_shared` is not in the Tier-1 contract baseline and therefore not tested or gated.

### Implementation pattern for runtime introspection

The existing implementation is correct and safe. The key pattern:

```rust
// SAFE: uses the shared LazyLock runtime, no new runtime created
let runtime = classic_shared_core::get_runtime();
let metrics = runtime.metrics();  // tokio::runtime::RuntimeMetrics
```

`tokio::runtime::Handle::metrics()` returns `tokio::runtime::RuntimeMetrics` which provides `num_workers()`, `num_alive_tasks()`, `active_tasks_count()` and similar fields. These are read-only observations of the existing runtime. No new runtime is created. The ONE RUNTIME RULE is not violated.

If `runtime_thread_count` (count of worker threads) is the specific helper requested: `num_workers()` is the correct tokio API call. If `is_runtime_active` (can schedule tasks): `block_on(async { true })` is the existing C++ bridge pattern (see `runtime.rs:is_runtime_active`).

### New deps needed

**No new dependencies.** `tokio` is already a workspace dep with `features = ["full"]`, which includes `runtime-metrics`. PyO3 0.27.2 is already the workspace dep. The implementation already exists; only parity gate enrollment is needed.

### Tokio metrics feature note

`tokio::runtime::RuntimeMetrics` (accessed via `Handle::metrics()`) requires the `tokio_unstable` cfg flag OR the `metrics` feature depending on the tokio version. In tokio 1.49.0 with `features = ["full"]`, `RuntimeMetrics` is available without `tokio_unstable` â€” `num_workers()` is stable. MEDIUM confidence on exact feature requirements for all metrics fields; `num_workers()` specifically is documented as stable. The existing implementation compiles today (it is in `classic-shared-py/src/lib.rs`), which is the ground truth.

## Finding 4: Node PE-Version Extraction

### Current state

- C++ bridge: `extract_pe_version_string(exe_path: &str) -> String` in `game.rs`, using `pelite::pe64::Pe::from_bytes`
- Python: `extract_pe_version(path: str) -> PeVersion` and `is_valid_pe_path(path: str) -> bool` in `classic_version.pyi`, backed by `classic-version-py`
- Node `version.rs`: No PE extraction. Has `parse_version`, `compare_versions`, `extract_version_from_filename`, `extract_version_from_log`, `extract_all_versions`, `format_version`. Deliberately omits PE extraction.

This is an active parity gap per `binding-parity-overview.md`: "C++ has PE probing that Node does not expose, while Node has broader semver-style helpers than C++ does."

### Implementation needed

Add to `ClassicLib-rs/node-bindings/classic-node/src/version.rs`:

```rust
/// Extract file version from a Windows PE executable or DLL.
/// Returns a 4-part tuple [major, minor, patch, build].
/// @throws if the path is invalid or the file has no version resource.
#[napi]
pub fn extract_pe_version(path: String) -> Result<Vec<u32>> { ... }

/// Check whether path points to a valid PE file (.exe or .dll).
#[napi]
pub fn is_valid_pe_path(path: String) -> bool { ... }
```

The `classic-version-core` crate already exports `extract_pe_version` returning `Result<(u32, u32, u32, u32)>` (confirmed from `game.rs` usage: `extract_pe_version(Path::new(exe_path))`). The NAPI wrapper converts the tuple to `Vec<u32>` for JS.

For a buffer-based API (where the caller already has bytes): `pelite` supports `PeFile::from_bytes(bytes: &[u8])` for in-memory PE parsing. This is a useful addition for Node consumers that hold a buffer. The caller passes `Buffer` (a NAPI `Uint8Array` or `Buffer`) and avoids a disk read. This mirrors `pelite`'s own API design.

### New deps needed

**No new dependencies.** `pelite 0.10` is already a workspace dependency (confirmed in `ClassicLib-rs/Cargo.toml`). The `classic-node` crate already depends on `classic-version-core` (confirmed in `ClassicLib-rs/node-bindings/classic-node/Cargo.toml`). No `pelite` direct dep is needed in `classic-node`; all pelite logic lives in `classic-version-core`, which is already imported.

### Buffer vs path API recommendation

Expose both for full parity with Python's `extract_pe_version(path)`:
- `extractPeVersion(path: string): [number, number, number, number]` â€” path-based, mirrors Python/C++ API
- `isValidPePath(path: string): boolean` â€” guard helper, mirrors Python's `is_valid_pe_path`

Buffer-based `extractPeVersionFromBuffer(buffer: Buffer)` is a Node-specific enhancement (no Python/C++ equivalent) and can be deferred unless a phase explicitly asks for it.

## Finding 5: Per-Binding Error Contract Documentation

### Problem

C++ uses `""` / `false` / sentinel DTOs. Node uses `null` or throws. Python raises typed exceptions. No single source of truth documents these conventions or links them back to Rust error variants.

### What exists today

- Python: Typed exception classes in `classic-shared-py/src/exceptions.rs` and `classic-shared-py/src/error_convert.rs`. The `ToPyErr` trait maps `ClassicError` variants to `PyRuntimeError`, `PyIOError`, `PyValueError`, etc. `.pyi` stubs document `Raises:` per function.
- Node: `to_napi_err()` helper function (local to each module) converts display strings to `napi::Error` with `code` field. Documented per-function in `///` TSDoc comments.
- C++: CXX converts Rust panics/exceptions to `rust::Error` caught in C++ as `catch (const rust::Error& e)`. No structured error type catalog.

### Tools considered

| Tool | Version | Status | Assessment |
|------|---------|--------|------------|
| `cargo-public-api` | 0.51.0 | External CLI tool | Useful for detecting public API changes (error types, variants) between commits. Does not generate language-specific error documentation. |
| `public-api` | 0.51.0 | Rust library | Same as above; library version of `cargo-public-api`. Useful in a CI diff script. |
| `schemars` | 1.2.1 | Rust crate | Generates JSON Schema from Rust types via `#[derive(JsonSchema)]`. Could document error shapes as schemas. Requires adding `#[derive(JsonSchema)]` to each error enum. Adds `schemars` as a workspace dep. |
| `rustdoc-json` | 0.9.9 | Rust crate | Utilities for working with rustdoc JSON output. Combined with `nightly` rustdoc, produces machine-readable API docs including error variants. No schema generation. |

### Recommendation

**Use `cargo-public-api` as an external CI tool (no workspace dep).** Install it in CI (`cargo install cargo-public-api`) and run it against `classic-shared-core`, `classic-scanlog-core`, `classic-config-core` to detect when public error types or variant sets change. This catches "Rust error changed without updating binding docs" cases.

**For documentation itself:** Write `docs/api/error-contract.md` (or extend `binding-parity-overview.md`) as a static mapping of each Rust error variant â†’ Python exception â†’ Node error code â†’ C++ exception/sentinel. This is authoring work, not tooling work. The parity contract JSON files already track function signatures; error shapes are a docs annotation, not a runtime-checked property.

**Do NOT add `schemars` to the workspace.** The overhead of annotating every error enum with `#[derive(JsonSchema)]` across 19 crates exceeds the value for this milestone. The error contract doc is a one-time authoring task.

### New deps needed

**No new workspace dependencies.** `cargo-public-api 0.51.0` is used as an external CLI tool in CI (`cargo install`), not a workspace dependency. No changes to `Cargo.toml` are needed.
