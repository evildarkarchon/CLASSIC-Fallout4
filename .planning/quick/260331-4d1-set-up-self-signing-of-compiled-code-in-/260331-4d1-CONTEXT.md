# Quick Task 260331-4d1: Self-signing of compiled code in build scripts - Context

**Gathered:** 2026-03-31
**Status:** Ready for planning

<domain>
## Task Boundary

Set up simple self-signing of final compiled code in the build scripts. Certificate lasts a year, only regenerated within a month of expiration.

</domain>

<decisions>
## Implementation Decisions

### Signing Tool
- Use `signtool.exe` from the Windows SDK — industry-standard for EXE/DLL signing, uses SignerSign API

### Certificate Storage
- PFX file on disk in a known/configurable location — portable, no cert store dependency, works across machines
- Must be `.gitignore`d to avoid leaking private keys

### What Gets Signed
- Sign EXEs only: `classic-cli.exe` (CLI) and `CLASSIC.exe` (GUI — CMake target name, not `classic-gui.exe`)
- `classic-cpp-bridge` is a static `.lib`, not a DLL — cannot and does not need signing (corrected from original discussion per research finding)
- Do NOT sign third-party DLLs (Qt, MSVC runtime, vcpkg dependencies)

### Claude's Discretion
- Exact PFX file path and naming convention
- Certificate subject/CN naming
- How the 30-day renewal window check is implemented
- Integration point in build scripts (after build vs after install vs after package)

</decisions>

<specifics>
## Specific Ideas

- Certificate should last 1 year
- Auto-regeneration should trigger only within 30 days of expiration
- Must integrate cleanly with existing `build_cli.ps1` and `build_gui.ps1` PowerShell scripts
- Should be opt-in (not block builds if signing tools or cert are unavailable)

</specifics>
