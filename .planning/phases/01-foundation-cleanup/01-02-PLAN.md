<task type="auto">
  <name>Task 1: Install vulture and run initial scan</name>
  <files>pyproject.toml</files>
  <action>
1. **Install vulture as dev dependency:** Run `uv add --dev vulture`. This updates pyproject.toml and uv.lock.

2. **Run initial vulture scan:** Run `uv run vulture ClassicLib/ --min-confidence 80` and capture the full output to a temporary file for analysis. Expect 30-100+ results, most of which will be false positives.

3. **Categorize every vulture finding** into three buckets:
   - **True dead code:** Functions/classes/variables that are genuinely unused anywhere. These should be removed.
   - **False positives (dynamic dispatch):** PySide6 slot methods connected via signals, pytest fixtures discovered by pytest, PyO3 module-level imports used at C level, `__all__` exports, click/argparse callbacks.
   - **Uncertain:** Items that need further investigation (grep for callers).

4. **Remove true dead code** identified by vulture. For each item:
   - Verify it has no callers: `grep -rn "function_name" ClassicLib/ tests/`
   - Remove the dead code
   - Run tests to confirm nothing breaks

5. **Generate whitelist for false positives:** Create `vulture_whitelist.py` in the project root. For each confirmed false positive, add an entry. The whitelist format is: import or reference the name so vulture sees it as "used."

   Structure the whitelist with clear sections:
   ```python
   """Vulture whitelist -- verified false positives for dead code detection.

   Each entry references a name that vulture cannot detect as used because it is
   invoked dynamically (Qt signals, pytest discovery, PyO3 C-level imports, etc.).
   """

   # PySide6 slots (connected via Qt signal-slot mechanism)
   # ...

   # Pytest fixtures (discovered by pytest, not imported directly)
   # ...

   # PyO3 imports (used at C/FFI level, not visible to static analysis)
   # ...

   # Other dynamic dispatch
   # ...
   ```

6. **Verify clean run:** Run `uv run vulture ClassicLib/ vulture_whitelist.py --min-confidence 80` and confirm 0 violations.

NOTE: Do not redirect output to `nul` or `NUL` on Windows. Capture to files if needed.
  </action>
  <verify>
- `uv run vulture ClassicLib/ vulture_whitelist.py --min-confidence 80` reports 0 violations (exit code 0)
- `vulture_whitelist.py` exists in project root with categorized sections
- `uv run pytest -m "unit and not slow" -x` still passes after any dead code removal
  </verify>
  <done>Vulture installed, initial scan complete, all true dead code removed, whitelist curated with only verified false positives, clean vulture run with 0 violations.</done>
</task>

  <action>
1. **Read the existing CI config** at `.github/workflows/ci.yml` to understand the current job structure, Python version setup, and caching patterns.

2. **Add a new job** named `dead-code` (or similar) to the CI workflow. Model it after the existing lint/test jobs for consistency. The job should:
   - Run on `windows-latest` (matching existing CI)
   - Check out the repository
   - Install uv and set up Python (same version as other jobs)
   - Run `uv sync --all-extras` to install dependencies including vulture
   - Run `uv run vulture ClassicLib/ vulture_whitelist.py --min-confidence 80`
   - Fail if vulture finds any violations (vulture returns non-zero exit code on findings)

3. **Add pyproject.toml vulture config** if not already present. Add under `[tool.vulture]`:
   ```toml
   [tool.vulture]
   paths = ["ClassicLib/"]
   exclude = ["**/tests/*", "**/.venv/*", "**/dist/*", "**/build/*"]
   min_confidence = 80
   ```

   Note: Even with pyproject.toml config, the CI command should explicitly pass the whitelist file path since vulture's pyproject.toml support for whitelist paths can be inconsistent.

4. **Verify the CI YAML is valid** by checking syntax (proper indentation, valid YAML structure).
