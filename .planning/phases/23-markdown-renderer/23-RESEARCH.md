# Phase 23: Markdown Renderer - Research

**Researched:** 2026-02-05
**Domain:** Markdown parsing and rich-text rendering in Slint GUI
**Confidence:** HIGH

## Summary

This phase transforms the existing plain-text `TextEdit`-based report viewer into a formatted markdown display. The current `ReportViewer` component in `report_viewer.slint` is a simple `TextEdit { read-only: true }` with Consolas font. Reports are markdown-formatted text (H1/H3 headings, bold, italic, blockquotes, horizontal rules, bullet lists, indented code blocks) generated by the Rust `ReportGenerator` in `classic-scanlog-core`.

The recommended approach is: (1) parse markdown with `pulldown-cmark` in Rust, (2) convert to a flat `Vec<MarkdownBlock>` intermediate representation, (3) render via a Slint `for` loop over a model of discriminated block structs with conditional visibility per block type. The `TextEdit` is replaced with a `ScrollView` containing a `VerticalLayout` with `for block in model` iteration. Each block type (heading, paragraph, code block, horizontal rule, list item) is a sub-component with appropriate styling.

**Primary recommendation:** Use pulldown-cmark to parse markdown into a flat block model, render via Slint `for` loop with type-discriminated structs and conditional `if` visibility for each block type.

## Architecture Patterns

### Recommended Architecture: Block Model Pattern

The core architecture is a three-layer pipeline:

```
Markdown String  -->  pulldown-cmark Parser  -->  Vec<MarkdownBlock>  -->  Slint VecModel  -->  for loop rendering
```

**Layer 1: Parsing (Rust, `src/markdown.rs`)**
Parse markdown string with pulldown-cmark into an intermediate `Vec<MarkdownBlock>`.

**Layer 2: Model Bridge (Rust to Slint)**
Convert `Vec<MarkdownBlock>` into a `VecModel<SlintMarkdownBlock>` that Slint can iterate over.

**Layer 3: Rendering (Slint, `ui/widgets/report_viewer.slint`)**
`for` loop iterates over the model. Each block type rendered via conditional `if` visibility.

### Recommended Project Structure
```
rust/ui-applications/classic-gui/
  src/
    markdown.rs          # NEW: pulldown-cmark parsing -> MarkdownBlock model
    results.rs           # MODIFIED: get_report_content -> get_report_blocks
    main.rs              # MODIFIED: wire up markdown model to Slint
  ui/
    widgets/
      report_viewer.slint   # REWRITTEN: ScrollView + for loop block rendering
      markdown_block.slint  # NEW: per-block-type rendering components
    main.slint              # MODIFIED: property types for markdown model
```

### Pattern 1: Type-Discriminated Block Model

**What:** A flat struct with an `int` type field used as discriminator, plus typed properties for each block variant.

**When to use:** Slint does not support Rust enums directly. Use an `int` type field with constants (0=paragraph, 1=heading, 2=code_block, etc.) and conditional `if` rendering per type.

**Slint struct definition:**
```slint
export struct MarkdownBlock {
    block-type: int,        // 0=paragraph, 1=heading, 2=code-block, 3=rule, 4=list-item, 5=blockquote
    text: string,           // Rendered text content (no markdown syntax)
    heading-level: int,     // 1, 2, or 3 for headings
    is-bold: bool,          // Whether text is bold
    is-italic: bool,        // Whether text is italic
    indent-level: int,      // Nesting depth for list items
    is-code: bool,          // Whether this is a code/monospace block
    bullet-marker: string,  // Bullet character for list items
}
```

**Slint rendering pattern:**
```slint
for block[idx] in root.blocks: VerticalLayout {
    // Heading
    if block.block-type == 1: Text {
        text: block.text;
        font-size: block.heading-level == 1 ? 22px :
                   block.heading-level == 2 ? 18px : 15px;
        font-weight: 700;
        wrap: word-wrap;
    }

    // Paragraph
    if block.block-type == 0: Text {
        text: block.text;
        font-size: 13px;
        wrap: word-wrap;
    }

    // Code block
    if block.block-type == 2: Rectangle {
        background: #2a2a2a;
        border-radius: 4px;
        // ... monospace text inside
    }

    // Horizontal rule
    if block.block-type == 3: Rectangle {
        height: 1px;
        background: #555555;
    }

    // List item
    if block.block-type == 4: HorizontalLayout {
        // indent + marker + text
    }
}
```

### Pattern 2: Raw Source Preservation for Copy

**What:** Store the original markdown source string separately from the parsed blocks.

**When to use:** The "Copy All" button must copy raw markdown source, while the viewer shows rendered content.

**Implementation:** The Rust side keeps both representations:
- `report-content` property: raw markdown string (for Copy All)
- `report-blocks` property: `[MarkdownBlock]` model (for rendering)

The Slint `ReportViewer` displays from `report-blocks` but Copy All reads from `report-content`.

### Pattern 3: ScrollView + VerticalLayout (Not ListView)

**What:** Use `ScrollView > VerticalLayout > for` instead of `ListView > for`.

**Why:** ListView virtualizes rows and requires fixed-height items. Markdown blocks have variable heights (headings, multi-line paragraphs, code blocks). ScrollView with VerticalLayout allows natural variable-height content. Report sizes are moderate (100-200 blocks max), so virtualization is unnecessary.

**Critical detail:** When using `for` inside a Flickable/ScrollView, `viewport-height` is NOT automatically calculated -- it must be computed. However, inside a VerticalLayout within a ScrollView, the layout computes its preferred-height automatically, and this works correctly as the viewport size.

### Anti-Patterns to Avoid
- **Do NOT use ListView for markdown blocks:** ListView requires fixed row heights and virtualizes content. Markdown blocks have variable heights.
- **Do NOT try to use Slint's experimental @markdown():** It's not stable in 1.15, provides no custom styling control, and would create upgrade risk.
- **Do NOT parse markdown on every render:** Parse once when report is selected. Store the block model. Only re-parse when a different report is selected.
- **Do NOT store HTML from pulldown-cmark:** The HTML output of pulldown-cmark is irrelevant -- we need structured blocks, not HTML.
- **Do NOT try to render inline formatting within a single Text element:** Slint Text does not support mixed bold/normal/italic within one element. Instead, flatten inline formatting: each paragraph becomes one block with the dominant formatting, or split into inline segments.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Markdown parsing | Custom regex-based parser | pulldown-cmark 0.13.0 | CommonMark edge cases (nested emphasis, escapes, entity refs) are complex. pulldown-cmark handles them all. |
| Clipboard access | Platform-specific clipboard code | arboard 3 (already in project) | Cross-platform clipboard is deceptively complex. Already working in project. |
| Text metrics/wrapping | Manual line-width calculation | Slint `wrap: word-wrap` on Text elements | Slint's text layout engine handles font metrics, wrapping, and Unicode correctly. |

**Key insight:** The pulldown-cmark parser handles all CommonMark edge cases (nested emphasis, escaped characters, entity references, indentation rules). A regex-based parser would miss these.

## Common Pitfalls

### Pitfall 1: Inline Formatting Complexity
**What goes wrong:** Trying to render mixed bold/italic/code within a single paragraph as one Slint `Text` element.
**Why it happens:** Slint `Text` supports only ONE font-weight, ONE font-italic setting per element. It cannot mix bold and normal text inline.
**How to avoid:** Two approaches:
1. **Simple (recommended for this phase):** Flatten inline formatting per block. A paragraph with `**bold**` text becomes a bold paragraph. Use the dominant formatting.
2. **Advanced:** Split paragraphs into inline segments (each a separate `Text` element in a `HorizontalLayout` with `wrap: word-wrap`). This adds complexity.
**Recommendation:** Start with approach 1 (simple flattening). The actual CLASSIC reports use bold/italic at the line level, not mid-sentence. The report format naturally aligns with block-level formatting.
**Warning signs:** Attempting to create a "rich text" element that mixes formatting within one Text.

### Pitfall 2: ScrollView Viewport Sizing with For Loops
**What goes wrong:** Scrollbar doesn't appear or content is clipped.
**Why it happens:** Slint's `Flickable` (underlying ScrollView) does NOT auto-calculate `viewport-height` when using a `for` loop (documented bug). With a VerticalLayout inside, it should auto-size from the layout's preferred height.
**How to avoid:** Place the `for` loop inside a `VerticalLayout` within the `ScrollView`. The VerticalLayout computes its height from children, and ScrollView uses that as viewport-height. Test with long reports to verify scrolling works.
**Warning signs:** Content cuts off, scrollbar is missing, or scrollbar thumb doesn't reflect content size.

### Pitfall 3: Performance with Large Reports
**What goes wrong:** UI freezes when selecting a report with many blocks.
**Why it happens:** Parsing a large report and creating 200+ Slint model entries on the UI thread.
**How to avoid:** Parse markdown and build the block model on the Rust side before pushing to Slint. The actual parsing is fast (microseconds for typical reports). The bottleneck would be Slint element instantiation, but 100-200 blocks is well within acceptable range.
**Warning signs:** Noticeable delay when selecting different reports.

### Pitfall 4: Copy All Behavior Regression
**What goes wrong:** Copy All copies rendered text instead of original markdown source.
**Why it happens:** The viewer switches from a TextEdit (which has `.text` = source) to a custom component (which shows rendered blocks). If Copy All reads from the rendered model, it loses markdown syntax.
**How to avoid:** Keep the raw markdown string as a separate property. The Copy All callback reads this property, not the rendered block model. This is the existing pattern (reads `report-content`).
**Warning signs:** Copied text lacks `#`, `**`, backticks.

### Pitfall 5: Blockquote and Indented Content Confusion
**What goes wrong:** pulldown-cmark treats 4-space indented content as code blocks per CommonMark spec, but CLASSIC reports use indentation for mod solution details.
**Why it happens:** The CLASSIC report format uses 4-space indentation for solution text (e.g., "Advised Fix: ..."), which CommonMark treats as a fenced code block.
**How to avoid:** Examine the actual pulldown-cmark output for sample reports. The report generator already uses 4-space indentation intentionally as code blocks. Verify with golden test files that the parsed structure matches expectations. If indented content should NOT be code blocks, the report generator in `classic-scanlog-core` may need adjustment (but this is likely out of Phase 23 scope).
**Warning signs:** Solution text appearing in monospace code block style instead of regular text.

## Code Examples

### pulldown-cmark Event Iteration
```rust
// Source: https://docs.rs/pulldown-cmark/0.13.0/pulldown_cmark/
use pulldown_cmark::{Event, Options, Parser, Tag, TagEnd, HeadingLevel, CodeBlockKind};

fn parse_markdown(source: &str) -> Vec<MarkdownBlock> {
    let parser = Parser::new(source); // Default CommonMark options
    let mut blocks = Vec::new();
    let mut current_text = String::new();
    let mut in_heading = false;
    let mut heading_level = 0;
    let mut in_code_block = false;
    let mut in_strong = false;
    let mut in_emphasis = false;
    let mut in_list_item = false;
    let mut list_depth = 0;

    for event in parser {
        match event {
            Event::Start(Tag::Heading { level, .. }) => {
                in_heading = true;
                heading_level = match level {
                    HeadingLevel::H1 => 1,
                    HeadingLevel::H2 => 2,
                    HeadingLevel::H3 => 3,
                    _ => 4,
                };
            }
            Event::End(TagEnd::Heading(_)) => {
                blocks.push(MarkdownBlock::heading(heading_level, &current_text));
                current_text.clear();
                in_heading = false;
            }
            Event::Start(Tag::Paragraph) => {}
            Event::End(TagEnd::Paragraph) => {
                if !current_text.is_empty() {
                    blocks.push(MarkdownBlock::paragraph(
                        &current_text, in_strong, in_emphasis,
                    ));
                    current_text.clear();
                }
                in_strong = false;
                in_emphasis = false;
            }
            Event::Start(Tag::CodeBlock(_kind)) => {
                in_code_block = true;
            }
            Event::End(TagEnd::CodeBlock) => {
                blocks.push(MarkdownBlock::code_block(&current_text));
                current_text.clear();
                in_code_block = false;
            }
            Event::Start(Tag::Strong) => { in_strong = true; }
            Event::End(TagEnd::Strong) => {} // stays bold for this block
            Event::Start(Tag::Emphasis) => { in_emphasis = true; }
            Event::End(TagEnd::Emphasis) => {}
            Event::Start(Tag::List(_)) => { list_depth += 1; }
            Event::End(TagEnd::List(_)) => { list_depth -= 1; }
            Event::Start(Tag::Item) => { in_list_item = true; }
            Event::End(TagEnd::Item) => {
                blocks.push(MarkdownBlock::list_item(
                    list_depth, &current_text, in_strong,
                ));
                current_text.clear();
                in_list_item = false;
                in_strong = false;
                in_emphasis = false;
            }
            Event::Start(Tag::BlockQuote(_)) => {}
            Event::End(TagEnd::BlockQuote(_)) => {
                if !current_text.is_empty() {
                    blocks.push(MarkdownBlock::blockquote(&current_text, in_strong));
                    current_text.clear();
                    in_strong = false;
                }
            }
            Event::Rule => {
                blocks.push(MarkdownBlock::rule());
            }
            Event::Text(text) => {
                current_text.push_str(&text);
            }
            Event::Code(code) => {
                // Inline code -- append with marker
                current_text.push('`');
                current_text.push_str(&code);
                current_text.push('`');
            }
            Event::SoftBreak => {
                current_text.push(' ');
            }
            Event::HardBreak => {
                current_text.push('\n');
            }
            _ => {} // Ignore HTML, footnotes, etc.
        }
    }
    blocks
}
```

### Slint Block Rendering Component
```slint
// Source: Pattern derived from Slint docs for conditional rendering
import { ScrollView, VerticalBox, HorizontalBox } from "std-widgets.slint";

export struct MarkdownBlock {
    block-type: int,     // 0=paragraph, 1=heading, 2=code-block, 3=rule, 4=list-item, 5=blockquote
    text: string,
    heading-level: int,
    is-bold: bool,
    is-italic: bool,
    indent-level: int,
    bullet-marker: string,
}

component MarkdownContent inherits VerticalLayout {
    in property <[MarkdownBlock]> blocks;
    spacing: 4px;
    padding: 8px;

    for block in root.blocks: VerticalLayout {
        // Heading block
        if block.block-type == 1: Text {
            text: block.text;
            font-size: block.heading-level == 1 ? 22px
                     : block.heading-level == 2 ? 18px
                     : 15px;
            font-weight: 700;
            wrap: word-wrap;
        }

        // Paragraph block
        if block.block-type == 0: Text {
            text: block.text;
            font-size: 13px;
            font-weight: block.is-bold ? 700 : 400;
            font-italic: block.is-italic;
            wrap: word-wrap;
        }

        // Code block
        if block.block-type == 2: Rectangle {
            background: #2a2a2e;
            border-radius: 4px;
            padding: 8px;

            code-text := Text {
                text: block.text;
                font-family: "Consolas";
                font-size: 12px;
                wrap: word-wrap;
                width: parent.width - 16px;
            }
            height: code-text.preferred-height + 16px;
        }

        // Horizontal rule
        if block.block-type == 3: Rectangle {
            height: 1px;
            background: #555555;
        }

        // List item
        if block.block-type == 4: HorizontalLayout {
            spacing: 6px;
            padding-left: block.indent-level * 16px;

            Text {
                text: block.bullet-marker;
                font-size: 13px;
                width: 16px;
                horizontal-alignment: center;
            }
            Text {
                text: block.text;
                font-size: 13px;
                font-weight: block.is-bold ? 700 : 400;
                wrap: word-wrap;
                horizontal-stretch: 1;
            }
        }

        // Blockquote
        if block.block-type == 5: HorizontalLayout {
            spacing: 8px;
            Rectangle {
                width: 3px;
                background: #555555;
                border-radius: 1px;
            }
            Text {
                text: block.text;
                font-size: 13px;
                font-weight: block.is-bold ? 700 : 400;
                font-italic: true;
                wrap: word-wrap;
                horizontal-stretch: 1;
            }
        }
    }
}
```

### Rust MarkdownBlock Model
```rust
// Intermediate representation between pulldown-cmark and Slint
#[derive(Clone, Debug)]
pub struct MarkdownBlock {
    pub block_type: i32,      // Matches Slint int constants
    pub text: String,
    pub heading_level: i32,
    pub is_bold: bool,
    pub is_italic: bool,
    pub indent_level: i32,
    pub bullet_marker: String,
}

// Block type constants
pub const BLOCK_PARAGRAPH: i32 = 0;
pub const BLOCK_HEADING: i32 = 1;
pub const BLOCK_CODE: i32 = 2;
pub const BLOCK_RULE: i32 = 3;
pub const BLOCK_LIST_ITEM: i32 = 4;
pub const BLOCK_BLOCKQUOTE: i32 = 5;

impl MarkdownBlock {
    pub fn heading(level: i32, text: &str) -> Self {
        Self {
            block_type: BLOCK_HEADING,
            text: text.to_string(),
            heading_level: level,
            is_bold: true,
            is_italic: false,
            indent_level: 0,
            bullet_marker: String::new(),
        }
    }

    pub fn paragraph(text: &str, bold: bool, italic: bool) -> Self {
        Self {
            block_type: BLOCK_PARAGRAPH,
            text: text.to_string(),
            heading_level: 0,
            is_bold: bold,
            is_italic: italic,
            indent_level: 0,
            bullet_marker: String::new(),
        }
    }

    pub fn code_block(text: &str) -> Self {
        Self {
            block_type: BLOCK_CODE,
            text: text.to_string(),
            heading_level: 0,
            is_bold: false,
            is_italic: false,
            indent_level: 0,
            bullet_marker: String::new(),
        }
    }

    pub fn rule() -> Self {
        Self {
            block_type: BLOCK_RULE,
            text: String::new(),
            heading_level: 0,
            is_bold: false,
            is_italic: false,
            indent_level: 0,
            bullet_marker: String::new(),
        }
    }

    pub fn list_item(depth: i32, text: &str, bold: bool) -> Self {
        let marker = match depth {
            1 => "\u{2022}",  // bullet
            2 => "\u{25E6}",  // open circle
            _ => "\u{25AA}",  // small square
        };
        Self {
            block_type: BLOCK_LIST_ITEM,
            text: text.to_string(),
            heading_level: 0,
            is_bold: bold,
            is_italic: false,
            indent_level: depth - 1,
            bullet_marker: marker.to_string(),
        }
    }

    pub fn blockquote(text: &str, bold: bool) -> Self {
        Self {
            block_type: BLOCK_BLOCKQUOTE,
            text: text.to_string(),
            heading_level: 0,
            is_bold: bold,
            is_italic: false,
            indent_level: 0,
            bullet_marker: String::new(),
        }
    }
}
```

## Inline Formatting Strategy

Given that Slint `Text` elements support only ONE font-weight and ONE font-italic per element, and the actual CLASSIC reports use inline formatting at roughly the paragraph/line level:

**Recommended approach: Block-level formatting flattening**

- A paragraph containing `**bold text**` becomes a bold paragraph
- A paragraph containing `*italic text*` becomes an italic paragraph
- A paragraph with mixed `**bold** and normal` text: use the first/dominant format
- Inline code (backticks) within paragraphs: keep as text with backtick characters visible

This works well because CLASSIC report lines are typically:
- All bold: `**Main Error:** Unhandled exception...`
- All italic: `*You have the latest version!*`
- Plain text: descriptions and credits
- Inline bold within list items: `**[!] FOUND : [21] Homemaker**`

The bold/italic state detected during parsing sets the whole block's formatting.

## Partial Text Selection Assessment

**Feasibility: LOW for custom Slint rendering**

Slint `Text` elements support text selection within a single element. However:
- The report is rendered as many separate `Text` elements (one per block)
- Cross-element selection (dragging from one `Text` to another) is NOT supported by Slint
- Only `TextEdit` and `TextInput` support selection natively

**Options:**
1. **Accept "Copy All" only** (user decision: fallback is acceptable)
2. **Use invisible TextEdit overlay** -- place a transparent `TextEdit { read-only: true }` with the rendered text on top, but this defeats the purpose of visual formatting
3. **Per-block selection** -- users can select text within a single rendered block (heading, paragraph, etc.)

**Recommendation:** Accept per-block selection naturally (Slint Text elements allow it within each element) and rely on "Copy All" for full-report copying. This matches the user's fallback preference.

## Open Questions

1. **Inline code pill rendering within paragraphs**
   - What we know: User wants inline code (backtick) to get a background pill highlight like GitHub/Discord.
   - What's unclear: Slint Text element cannot have partial background highlighting. A single `Text` element is either all-monospace-with-background or all-regular.
   - Recommendation: For inline code that appears within a paragraph, either (a) render the entire paragraph as a code-styled block if it contains mostly code, or (b) split the paragraph into segments with alternating regular/code Text elements in a HorizontalLayout with wrap. Option (b) is complex. Option (a) is simpler. Given that inline code in CLASSIC reports is rare (mainly backtick filenames), consider displaying backtick-wrapped text inline without the pill for now, and add pill rendering as a polish item.

2. **4-space indented content as code blocks**
   - What we know: pulldown-cmark treats 4-space indented text as code blocks (per CommonMark spec). CLASSIC reports use 4-space indentation for mod solution text.
   - What's unclear: Whether users expect solution text to appear as code blocks or as regular indented text.
   - Recommendation: Test with actual report content. If the code-block rendering looks reasonable for solution text (monospace with background), keep it. If it looks wrong, consider adding a post-processing step to convert code blocks that follow certain patterns back to regular indented text.

3. **Report content is line-based, not paragraph-based**
   - What we know: `AnalysisResult.report_lines` is `Vec<String>` where each line already includes `\n`. The lines are joined with `\n` for display.
   - What's unclear: Some lines may contain multiple logical markdown elements.
   - Recommendation: Join lines into a single markdown string, then parse with pulldown-cmark. The parser handles the complete markdown document structure correctly regardless of original line boundaries.
