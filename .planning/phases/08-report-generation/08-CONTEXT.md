# Phase 8: Report Generation - Context

**Gathered:** 2026-02-03
**Status:** Ready for planning

<domain>
## Phase Boundary

Wire Rust ReportGenerator/ReportComposer to Python wrapper. All report sections (header, error, suspect, settings, footer) generated by Rust — Python only calls Rust and writes the result to disk. VR mode is being deprecated and removed as part of this phase.

</domain>

<decisions>
## Implementation Decisions

### Format parity strictness
- Character-for-character parity required between Python and Rust output
- Normalize all paths to forward slashes for cross-platform consistency
- Mask timestamps with {{TIMESTAMP}} placeholders in golden files (consistent with Phase 6)
- Hard fail on parity mismatch — test fails immediately with diff output

### VR mode removal
- VR mode detection and indicator are being REMOVED entirely
- VR (Fallout4VR, SkyrimVR) is just another game variant — no special-casing
- Silent removal — no deprecation warnings needed (internal change)
- This supersedes roadmap success criterion #3 ("VR mode indicator appears correctly")

### Error presentation
- Show technical details for errors (helps debugging)
- Hard fail, no fallback — Rust error propagates to user (consistent with Phase 7)
- Include full context in errors: file paths, line numbers, Rust stack context
- Sections are optional based on log content (not error states)

### Section content and output
- Keep current section structure: header, error, suspect, settings, footer
- Include CLASSIC version information in report headers
- Output report in same location as input crash log
- Filename format: `<log_basename>-AUTOSCAN.md`

### Claude's Discretion
- Exact StringPool optimization approach in ReportComposer
- Internal error message formatting details
- Section ordering within constraints

</decisions>
