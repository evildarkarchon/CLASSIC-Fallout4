# Phase 8: Report Generation - Research

**Researched:** 2026-02-03
**Domain:** Rust ReportGenerator/ReportComposer wiring to Python, VR removal, format parity validation
**Confidence:** HIGH

## Summary

Phase 8 focuses on wiring the existing Rust `ReportGenerator` and `ReportComposer` classes to Python such that all report sections are generated by Rust, with Python only calling Rust and writing output to disk. The Rust implementation is already complete (90-100%) in `classic-scanlog-core` and exposed via `classic-scanlog-py`. The primary work involves:

1. **Thinning Python wrappers** - Remove Python fallback logic from `RustAcceleratedReportGenerator` (currently has Python fallback code)
2. **VR mode removal** - Remove VR detection indicators and special-casing from report generation (VR is just another game variant)
3. **Parity validation** - Ensure character-for-character parity between current Python output and new Rust-only output
4. **Path normalization** - Convert all paths to forward slashes for cross-platform consistency

The Rust implementation uses StringPool for memory-efficient string interning and supports parallel fragment composition via Rayon. All section generators (header, error, suspect, settings, footer) exist in Rust and are accessible via PyO3 bindings.

**Primary recommendation:** Remove Python fallback paths from `ClassicLib/integration/rust/report/generator.py` and `ClassicLib/scanning/logs/report_generator.py`, wire all section generation through Rust `ReportGenerator`, and eliminate VR-specific report indicators.

**What:** Use Rust StringPool for repeated strings in report generation
**When to use:** All fragment creation for memory efficiency
**Example:**

### Anti-Patterns to Avoid

- **Python fallback paths:** Remove all `else: use_python_implementation()` branches
- **VR special-casing:** No `is_vr_log` parameters or VR indicators in headers
- **Mutable state:** Use immutable `ReportFragment` composition, not mutable lists
- **Mixed Rust/Python fragment types:** All fragments should be Rust `PyReportFragment`

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| String deduplication | Custom string caching | `StringPool` with `string_cache::DefaultAtom` | Thread-safe, proven performance |
| Parallel composition | Manual thread management | `ReportComposer.compose_parallel()` with Rayon | Automatic parallel reduction |
| Fragment concatenation | List append loops | `ReportFragment.combine()` | Immutable, Arc-based efficiency |
| Report assembly | Manual string building | `ReportComposer.build_string()` | Pre-computed capacity, efficient newline handling |

**Key insight:** The Rust implementation already handles all the complexity (parallel processing, memory pooling, efficient string building). Python just needs to call Rust methods and write the result.

## Common Pitfalls

### Pitfall 1: API Signature Mismatch

**What goes wrong:** Rust and Python APIs have different signatures (Rust `generate_error_section` takes `is_outdated: bool`, Python takes multiple version parameters)
**Why it happens:** The Python wrapper currently does version comparison logic before calling Rust
**How to avoid:** Keep version comparison in Python, pass only the computed `is_outdated` boolean to Rust
**Warning signs:** Type errors or unexpected function arguments during testing

### Pitfall 2: VR Removal Breaking Tests

**What goes wrong:** Existing tests may check for VR indicators in reports
**Why it happens:** Tests captured expectations with VR mode behavior
**How to avoid:** Update golden files and test expectations simultaneously with VR removal
**Warning signs:** Test failures mentioning "VR MODE AUTO-DETECTED" strings

### Pitfall 3: Fragment Type Confusion

**What goes wrong:** Mixing Python `ReportFragment` (dataclass) and Rust `PyReportFragment` (PyO3 class)
**Why it happens:** Multiple fragment implementations exist in codebase
**How to avoid:** Always use `RustAcceleratedReportFragment` wrapper which handles both transparently
**Warning signs:** `AttributeError` on fragment objects (different methods: `has_content` vs `is_empty()`)

### Pitfall 4: Newline Inconsistency

**What goes wrong:** Character-for-character parity fails due to trailing newlines
**Why it happens:** Rust `build_string()` adds newlines; Python may double-add
**How to avoid:** Use single code path (Rust `build_string()`) for final output
**Warning signs:** Diff showing extra blank lines at section boundaries

### Pitfall 5: Path Separator Platform Differences

**What goes wrong:** Windows backslashes vs Unix forward slashes in paths
**Why it happens:** File paths captured differently per platform
**How to avoid:** Normalize all paths to forward slashes before embedding in report (per decision)
**Warning signs:** Golden file tests passing on one OS, failing on another

        return RustAcceleratedReportFragment.wrap_fragment(fragment, self._use_rust)
