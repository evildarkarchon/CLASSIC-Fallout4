## Objective

Create the GitHub Actions workflow foundation for automated benchmark regression detection. This plan establishes the core infrastructure: running benchmarks in CI, caching baselines between runs, and updating baselines when code merges to main.

## Context

Phase 13 established Criterion benchmark infrastructure with quick/thorough modes (50/200 samples), baseline management via critcmp, and benchmark scripts. This plan builds CI integration on top of that foundation.

**User decisions (LOCKED):**
- Store baselines in GitHub Actions cache (not committed to repo)
- Run all benchmarks (full suite, not subset)
- Same quick mode (50 samples) for both PRs and main branch
- Trigger only when PR is marked ready for review
- Baselines update automatically on main branch merge only

## Tasks

<task id="1">
Create benchmark workflow file with event triggers

Create `.github/workflows/benchmarks.yml` with:
- Triggers: `pull_request: types: [ready_for_review]` for main branch, `push` to main
- Concurrency group to prevent parallel runs for same PR
- Environment variables: `BENCH_MODE=quick`, `CARGO_TERM_COLOR=always`
- Permissions: `contents: read`, `pull-requests: write`
- Timeout: 60 minutes
- Skip condition: `if: github.event_name == 'push' || !github.event.pull_request.draft`

Reference existing ci.yml for patterns (Rust toolchain, Python setup, cargo caches).
</task>

<task id="2">
Add benchmark execution steps

Add steps to the workflow:
1. Checkout code (actions/checkout@v4)
2. Install Rust toolchain (dtolnay/rust-toolchain@stable)
3. Set up Python 3.12 (actions/setup-python@v5) - required for PyO3 builds
4. Cache cargo registry and build (reuse patterns from ci.yml)
5. Install critcmp: `cargo install critcmp --locked`
6. Run benchmarks in quick mode: `cargo bench --manifest-path rust/Cargo.toml -- --save-baseline current`

The benchmarks must complete within 60 minutes. Use `working-directory: rust` where appropriate.
</task>

<task id="3">
Implement baseline cache restore and save

Add cache steps using actions/cache@v4:

For **PR builds** (restore only):
```yaml
- name: Restore baseline cache
  id: baseline-cache
  uses: actions/cache/restore@v4
  with:
    path: rust/target/criterion/baseline
    key: criterion-baseline-${{ runner.os }}-dummy  # Never matches, forces restore-keys
    restore-keys: criterion-baseline-${{ runner.os }}
```

For **main branch builds** (save after benchmarks):
```yaml
- name: Copy current to baseline (main only)
  if: github.ref == 'refs/heads/main'
  working-directory: rust
  shell: bash
  run: |
    if [ -d "target/criterion/current" ]; then
      rm -rf target/criterion/baseline
      cp -r target/criterion/current target/criterion/baseline
    fi

- name: Save baseline cache (main only)
  if: github.ref == 'refs/heads/main'
  uses: actions/cache/save@v4
  with:
    path: rust/target/criterion/baseline
    key: criterion-baseline-${{ runner.os }}-${{ github.sha }}
```

Key design: PRs restore existing baseline but never save. Main branch runs always save a new baseline keyed by commit SHA. This ensures PRs always compare against the latest main branch baseline.
</task>

<task id="4">
Create benchmark threshold configuration file

Create `rust/benchmark-config.yaml` with default and per-benchmark thresholds:

```yaml
# Benchmark regression detection thresholds
# Used by CI to determine warning vs failure states
# This file is READ by the threshold analysis step in .github/workflows/benchmarks.yml

defaults:
  warning_threshold: 5   # 5% regression = warning annotation
  failure_threshold: 10  # 10% regression = build failure

# Per-benchmark overrides for noisy benchmarks
# Keys must match benchmark names exactly as they appear in critcmp output
overrides:
  # Example:
  # yaml_parsing/parse/5000_lines:
  #   warning_threshold: 8
  #   failure_threshold: 15
  #   reason: "Large YAML parsing has high variance on shared runners"
```

**IMPORTANT:** This file MUST be read by the threshold analysis logic in Plan 17-02 Task 1. The analysis script must:
1. Parse this YAML file at runtime
2. Use `defaults.warning_threshold` and `defaults.failure_threshold` as base values
3. Apply per-benchmark overrides from the `overrides` section when benchmark name matches
</task>

<task id="5">
Add baseline existence check and skip logic

Add step to detect missing baseline (first run scenario):

```yaml
- name: Check baseline exists
  id: baseline-check
  working-directory: rust
  shell: bash
  run: |
    if [ -d "target/criterion/baseline" ]; then
      echo "baseline_exists=true" >> $GITHUB_OUTPUT
      echo "Baseline found - comparison will be performed"
    else
      echo "baseline_exists=false" >> $GITHUB_OUTPUT
      echo "::warning::No baseline found - skipping comparison. Run benchmarks on main branch first."
    fi
```

This output is used by Plan 17-02 to skip comparison when no baseline exists (pass with warning annotation).
</task>
