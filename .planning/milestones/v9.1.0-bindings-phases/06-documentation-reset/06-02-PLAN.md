**B) Create `docs/api/binding-parity-policy.md` (per D-14, DOC-06).**

New file. Structure:

1. **Title**: "# Binding Parity Policy"
2. **One-Tier Policy Statement**: All Rust public API symbols in business-logic `-core` crates are exposed through all three binding surfaces (C++, Node, Python). There is no deferred tier, no backlog tier, no graduated promotion. When a new `pub fn` or `pub struct` is added to a `-core` crate's `lib.rs`, all three bindings must expose it before CI passes.
3. **Gate Ownership**:
   - CXX gate: `tools/cxx_api_parity/check_parity_gate.py` -- owns C++ bridge surface. Baseline: `tools/cxx_api_parity/cxx_baseline_surface.json`. Contributor docs: `docs/api/cxx-parity-gate.md`.
   - Python gate: `tools/python_api_parity/check_parity_gate.py` -- owns Python binding surface. Baseline: `docs/implementation/python_api_parity/baseline/parity_contract.json`.
   - Node gate: `tools/node_api_parity/check_parity_gate.py` -- owns Node binding surface. Baseline: `docs/implementation/node_api_parity/baseline/parity_contract.json`.
4. **When Gates Refresh**: Gates refresh when the baseline contract drifts from the live Rust surface. Each gate has a `generate_baseline.py` (or equivalent) that regenerates the baseline. Run the gate first to detect drift, then regenerate if drift is intentional.
5. **How To Add a New Public Rust API** (step-by-step workflow):
   1. Add the `pub fn`/`pub struct` to the `-core` crate's `lib.rs`
   2. Add the C++ bridge entry in `classic-cpp-bridge/src/` (CXX shared struct if DTO, opaque type if stateful)
   3. Add the Node NAPI wrapper in `classic-node/src/` (NAPI auto-converts `snake_case` to `camelCase`)
   4. Add the Python PyO3 wrapper in `classic-*-py/src/` and update the `.pyi` stub
   5. Regenerate all three baselines: `generate_baseline.py` for each gate
   6. Run all three gates to verify zero drift
   7. Commit all changes in the same PR
6. **Reference**: `AGENTS.md`, `binding-parity-overview.md`.

**C) Create `docs/api/error-contract.md` (per D-16, D-17, D-18, HARM-05).**

New file. Structure:

1. **Title**: "# Error Contract Conventions"
2. **Scope**: Documents the per-binding error shape conventions for Rust `ClassicError` variants as they cross FFI boundaries. These conventions are intentional design choices, not inconsistencies to fix. Reference: `AGENTS.md`.
3. **C++ (CXX Bridge)** section (per D-16):
   - Pattern: `rust::Error` exceptions for hard failures, empty-string sentinels for fail-soft returns
   - Example 1: `db_pool_get_entry()` in `classic-cpp-bridge/src/database.rs` returns `""` on lookup failure because Qt callers check `.isEmpty()` rather than catching exceptions
   - Example 2: `db_pool_get_entry_typed()` in the same file returns a `FormIdEntryDto` with `found: false` for lookup misses. The DTO has fields `{formid, plugin, value, found}` where `found` is a `bool` -- it is NOT a status string. C++ callers check `dto.found` before using `dto.value`.
   - Example 3: Functions like `orchestrator_process_log()` in `scanner.rs` propagate `rust::Error` for infrastructure failures (runtime not initialized, config not loaded). Note: there is no `run_scan()` function in scanner.rs -- the scan entry points are `orchestrator_process_log`, `orchestrator_process_logs_batch`, and `orchestrator_process_logs_batch_with_progress`.
   - Source link: `ClassicLib-rs/cpp-bindings/classic-cpp-bridge/src/database.rs`, `ClassicLib-rs/cpp-bindings/classic-cpp-bridge/src/scanner.rs`
4. **Node (NAPI-RS)** section (per D-16):
   - Pattern: `napi::Error` with a `code` field matching the Rust error variant name (e.g., `"InvalidArg"`, `"ParseError"`)
   - Example 1: `config_error_to_napi_err()` in `classic-node/src/config.rs` converts `ConfigError` variants to NAPI errors with structured codes
   - Example 2: `settings_error_to_napi_err()` converts `SettingsError` variants
   - Example 3: `runtime_to_napi_err()` handles `anyhow::Error` wrapping with automatic downcast to typed errors
   - Source link: `ClassicLib-rs/node-bindings/classic-node/src/config.rs`
5. **Python (PyO3)** section (per D-16):
   - Pattern: Typed Python exception classes (e.g., `RustConfigParseError`, `RustConfigIOError`) with message inspection
   - Example 1: `config_error_to_pyerr()` in `classic-config-py/src/lib.rs` maps each `ConfigError` variant to a specific Python exception class
   - Example 2: `classic-shared-py` provides `define_exceptions!` and `register_exceptions!` macros plus the `ToPyErr` trait for consistent exception wiring across all Python binding crates
   - Example 3: Tests use `pytest.raises(RustConfigParseError)` with message inspection
   - Source link: `ClassicLib-rs/python-bindings/classic-config-py/src/lib.rs`, `ClassicLib-rs/foundation/classic-shared-py/src/lib.rs`
6. **Why They Differ** section (per D-17):
   - C++ uses empty-string sentinels because Qt callers (`classic-gui/`) are written to check `.isEmpty()` and display "not found" UI -- changing to exceptions would break existing call sites
   - Node uses `error.code` strings because the Node ecosystem convention is `catch (e) { if (e.code === "ParseError") ... }` -- this is idiomatic for JS/TS consumers
   - Python uses typed exception classes because the Python ecosystem convention is `except RustConfigParseError as e:` with `isinstance` checks -- this is idiomatic for Python consumers
   - The Out of Scope section in `REQUIREMENTS.md` explicitly excluded standardizing these shapes: "Intentional design -- Qt fail-soft callers depend on empty-string sentinel return"
7. **Conversion Helper Reference** (per D-18):
   - C++: No dedicated helper -- `block_on()` + `match` on `Result` in each bridge function
   - Node: `config_error_to_napi_err()`, `settings_error_to_napi_err()`, `runtime_to_napi_err()` in `classic-node/src/config.rs` (link)
   - Python: `define_exceptions!`, `register_exceptions!`, `ToPyErr` trait, `ResultExt`, `without_gil` in `classic-shared-py/src/lib.rs` (link); per-module converters like `config_error_to_pyerr()` in `classic-config-py/src/lib.rs` (link)
