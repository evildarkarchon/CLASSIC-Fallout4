    **A.2. Helper semantics (spec) — revised to match actual schemas:**

    The helper performs the following transformations, all idempotent. Each step explicitly tracks whether it applies to Python contracts, Node contracts, or both.

    1. **[BOTH] JSON load with order preservation** — `json.load(f, object_pairs_hook=dict)`. Python 3.7+ dicts are insertion-ordered so this preserves key order automatically. Helper script adds a defensive `assert sys.version_info >= (3, 7), "requires Python 3.7+ for dict ordering"` near the top (Round 2 Claude LOW fix).

    2. **[BOTH] Top-level `ownerModules` merge (Round 1 concern 15):**
       - If `ownerModules[<source>]` exists: remove that entry entirely. Do NOT write its description anywhere else; the target owner is broader and its existing description is authoritative.
       - **Missing-target rule, per-contract branching (Round 2 HIGH-3 / concern-15 fix — Codex; Round 3 Codex MEDIUM disambiguation):** the helper MUST NOT use a single uniform "missing target" rule. Branch by contract type (detected by row-field presence — see "Contract type detection" bullet below):
         - **Python contract**: if `ownerModules.settings` is missing, abort with `sys.exit(1)` and a diagnostic — this would indicate a corrupted Python contract baseline since `settings` is a known live entry. (At revision time, Python contract has BOTH `ownerModules.yaml` AND `ownerModules.settings`, so this branch never fires for Phase 1 — it is a corruption guard.)
         - **Node contract**: if `ownerModules.settings` is missing (the live case at revision time — Node `ownerModules` only has `scanlog`, `config`, `version_registry`, `aux`), emit a WARNING to stderr (`WARNING: Node contract top-level ownerModules has neither "yaml" nor "settings"; row-level reparenting will proceed but top-level metadata remains minimal`) and continue without editing the top-level map. Do NOT abort — subsequent row-level edits still run. Do NOT invent a `settings` entry in the Node top-level map — that is a contract-metadata decision out of scope for this helper.
       - **Contract type detection:** the helper distinguishes Python vs Node by row-field presence, NOT by filename. If any row in `tier1Mappings` has a `pythonModule` or `pythonExportPath` key, treat as Python. If any row has a `nodeExport` key, treat as Node. (At revision time both contracts are unambiguous; Python and Node row schemas are disjoint.)

    3. **[NODE ONLY] Squad metadata update (concern 15):**
       - For every `squads[<squad_name>].ownerModules` array found: if the array contains `<source>`, either remove the source if `<target>` is already present, or replace `<source>` with `<target>` otherwise. Preserve order.
       - At revision time, Node contract `squads.scanlog_config.ownerModules = ["scanlog", "config"]` and `squads.version_registry_aux.ownerModules = ["version_registry", "aux"]`. Neither contains `yaml`, so this step is a no-op for Phase 1. Kept for Phase 2/3 reuse.
       - Python contract has no `squads` block at revision time; step is a no-op there.

    4. **[BOTH] Tier mapping rows — schema-detected field names (Round 2 HIGH-3 fix — Codex):**

       For each row in the top-level tier-mapping array (name varies by contract: `tier1Mappings` in both Python and Node at revision time):

       - If `row.get("ownerModule") == <source>`: set it to `<target>`.
       - If `row.get("rustCrate") == <rust_crate_old>`: set it to `<rust_crate_new>`.
       - **Schema detection for the binding module field:** check which of these keys the row has (they never coexist — Python vs Node schemas are disjoint):
         - **Python row** (has key `pythonModule`): if `row["pythonModule"] == <binding_module_old>`, set it to `<binding_module_new>`.
         - **Python row** (has key `pythonExportPath`): if `row["pythonExportPath"].startswith(<binding_module_old> + ".")`, replace the prefix with `<binding_module_new> + "."`. Example: `"classic_yaml.YamlOperations.__init__"` -> `"classic_settings.YamlOperations.__init__"`.
         - **Node row** (has key `nodeExport`): the `nodeExport` value is a simple JS export name like `"parseYaml"`, NOT a dotted path. Do NOT rewrite it — leave it as-is. The Round 1 spec incorrectly assumed Node rows had a module-qualified path; they do not. The Round 2 Codex review confirmed this via direct file inspection.

       - **IMPORTANT:** The helper never rewrites `rustSymbol` or `id` fields. `rustSymbol` is the Rust-side identifier and is owner-agnostic. `id` is a row identifier (e.g., `"yaml-yaml-operations-class"`) that MAY contain the source owner as a prefix but rewriting it would break cross-references in diff reports. Row IDs are intentionally left alone and their stability is part of the contract design — the gate matches rows by `id` not by derived-from-owner slugs.
