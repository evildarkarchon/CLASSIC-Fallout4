# Requirements: CLASSIC Rust Migration v8.2.0-part2

**Defined:** 2026-02-02
**Core Value:** Python is the UI, Rust is the engine — every piece of business logic lives in Rust `-core` crates, Python only handles presentation and user interaction.

## v8.2.0-part2 Requirements

Requirements for Rust migration milestone. Each maps to roadmap phases.

### Scanning Orchestration (ORCH)

- [x] **ORCH-01**: HybridOrchestrator routes all single-log processing through Rust OrchestratorCore
- [x] **ORCH-02**: HybridOrchestrator routes all batch processing through Rust with unbounded parallelism
- [x] **ORCH-03**: VR mode is auto-detected per-log in Rust orchestrator
- [x] **ORCH-04**: Python OrchestratorCore wrapper reduced to thin delegation (< 100 lines) *(EXCEEDED: removed entirely)*
- [x] **ORCH-05**: All analyzer components (Plugin, FormID, Suspect, Mod, Record, Settings) called from Rust

### Game Detection (GAME)

- [x] **GAME-01**: GamePathFinder in Rust used for all path detection (registry, XSE log, cache)
- [x] **GAME-02**: Python game_path.py reduced to thin wrapper around Rust GamePathFinder
- [x] **GAME-03**: GlobalRegistry integration uses Rust-detected paths
- [x] **GAME-04**: XSE/ENB integrity checking delegated to Rust validators

### Report Generation (REPT)

- [x] **REPT-01**: All report sections generated by Rust ReportGenerator
- [x] **REPT-02**: Python report_generator.py reduced to calling Rust and writing output
- [x] **REPT-03**: VR mode indicator correctly appears in Rust-generated headers *(superseded by VR removal decision)*
- [x] **REPT-04**: ReportFragment composition uses Rust ReportComposer with StringPool

### Settings Management (SETT)

- [x] **SETT-01**: All YAML settings loading uses Rust classic-settings-core cache
- [x] **SETT-02**: Python YamlCache wrapper delegates to Rust DashMap-based cache
- [x] **SETT-03**: Batch settings loading uses Rust parallel tokio::join! pattern
- [x] **SETT-04**: Cache invalidation correctly clears Rust cache on settings update
- [x] **SETT-05**: Settings writing remains in Python (Rust cache is read-only)

### Parity Validation (VAL)

- [x] **VAL-01**: Golden file tests capture Python output for 10+ representative crash logs
- [x] **VAL-02**: Rust scanning output matches Python output character-for-character
- [x] **VAL-03**: Rust report generation matches Python report format exactly *(identified differences - expected)*
- [x] **VAL-04**: Rust game detection returns identical paths as Python implementation
- [x] **VAL-05**: All existing tests pass with Rust as primary code path

### Integration & Cleanup (INTG)

- [x] **INTG-01**: Factory functions return Rust types directly (no fallback branches)
- [x] **INTG-02**: Python business logic files removed after Rust replacement validated
- [x] **INTG-03**: PyInstaller build includes all Rust modules and passes smoke test
- [x] **INTG-04**: GUI functionality unchanged (AsyncBridge integration preserved)
- [x] **INTG-05**: CLI functionality unchanged with Rust code paths

## Out of Scope

Explicitly excluded. Documented to prevent scope creep.

| Feature | Reason |
|---------|--------|
| GUI framework migration | PySide6/Qt stays — GUI is Python's domain |
| New user-facing features | Migration only, no new capabilities |
| Rust GUI (egui/iced) | Out of scope per milestone decision |
| Settings persistence in Rust | Python owns YAML writes, Rust is read-only cache |
| i18n/localization | Future scope, stick to English strings |
| HTML/RTF report output | Markdown is standard, no format expansion |
| Steam API integration | Scope creep, rely on registry/XSE log |
| Performance benchmarking | Correctness first, optimization later |
